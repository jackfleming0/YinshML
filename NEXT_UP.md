## Decision gate #1 — after polish sprint + probes

**Prerequisite — polish-sprint bake-off.** Before walking into the gate, run a head-to-head tournament between a checkpoint trained *after* Track A lands and a pre-polish baseline. This is the only way to answer "did all that polish work actually move the needle, or did we just churn?" The infrastructure already exists (`yinsh_ml/utils/tournament.py::ModelTournament._play_match` + the Wilson-CI + deterministic-eval tooling we just landed). Concrete setup:

- **Baseline anchor.** Pin one of the recent pre-polish iteration checkpoints (e.g., the iter-6 peak from the 10-iter warm-start run at ELO +61 over seed) as the reference opponent. Copy it out of its training dir into a stable `models/baselines/` path so retention policies don't delete it.
- **Challenger.** The best checkpoint to come out of the polish sprint — typically the highest-ELO iteration after Tier-1 and the diagnostic probes have landed. If we want to isolate contributions, re-train short runs ablating one item at a time (Tier-1 minus subtree reuse, Tier-1 minus soft value targets, etc.) and bake-off each ablation.
- **Match format.** 200 games per direction (400 total), `eval_seed` pinned, EMA weights on both sides. Wilson 95% CI + straddle check reuses the promotion-gate machinery. Anything under ~53% win rate (lower CI ≤ 50%) is "polish didn't statistically beat baseline — re-scope."
- **Output.** A single `bakeoff_report.md` per comparison: win rate + CI + per-pair CIs + sample game IDs, plus an ELO-delta estimate via the Glicko tracker. Commit under `analysis_output/bakeoffs/`.

Ship this as a new `scripts/run_bakeoff.py` that takes two checkpoint paths and hands off to `ModelTournament`. Small script (~50-80 lines), mostly config glue.

Outcome determines the next phase. Three branches:

- **Branch A: §5 breaks the plateau (discrimination 0.104 → >0.13).** The plateau was a training-signal problem. AZ recipe is correct. Push `TODO_baseline.md` Tier 3-4 items and frontier items are mostly deferred. When GPU lands, scale up the polished AZ rather than pivoting.

- **Branch B: §5 fails to move it; §8 shows missing concepts.** Plateau is representational, not architectural. Add targeted auxiliary heads / features informed by SAE findings (positional threat, history planes, phase-conditioned heads). Still within AZ recipe but data-augmented. GPU planning is about capacity/depth, not architecture replacement.

- **Branch C: §5 fails; §8 shows rich internal representations.** Plateau is architectural. Commit to MuZero (`TODO_frontier.md` §1) or Transformer backbone (§2) when GPU lands. *Don't start either on Mac Mini* — wait for hardware.

§4's result is informative in every branch: if IQL warm-start survives iter_1 where BC fails, the supervised-vs-self-play training-regime gap is the real story. That conclusion holds regardless of §5 / §8.

### 2026-04-16

- [x] **Move encoder collision-free rework (Track A #1, final item).** Legacy encoding `((src*31 + dst) % 5848)` hashed 85×84 = 7140 (src, dst) ring-move pairs into 5848 slots but actually only reached 2687 distinct values (62% collision rate, 3161 slots structurally unreachable). The network couldn't learn to distinguish between distinct moves that shared a policy-head slot in MCTS training targets — quietly capping policy-head expressiveness for the whole history of this codebase. Replaced with direct `src_idx·(num_positions-1) + adjusted_dst_idx` (where `adjusted_dst` skips the src==dst diagonal), filling exactly 7140 slots with zero collisions. `index_to_move` inversion drops its old quadratic search loop for O(1) `src = idx // 84; adjusted = idx % 84; dst = adjusted if adjusted < src else adjusted + 1`. New layout (total 8390 slots): placement [0, 85), ring movement [85, 7225), ring removal [7225, 7310), marker removal [7310, 8390). `StateEncoder.total_moves` is now the single source of truth — `YinshNetwork.__init__` reads it via `StateEncoder().total_moves` instead of duplicating the constant, preventing future layout changes from going out of sync. Marker removal kept at its 1080-slot sequence hash; the 5-in-a-row action space isn't a simple Cartesian product and making it collision-free would need board-geometry enumeration (separate scope). **Checkpoint-compat handling**: total_moves changed from 7395 → 8390, so old policy-head tensors won't load into the new network. Added an explicit shape check in `NetworkWrapper.load_model` that raises `Policy-head size mismatch: checkpoint has 7395 ... current encoder emits 8390 ... retrain from scratch or migrate the policy head explicitly` — without this the silent shape-filter below would have randomly re-initialized the policy output layer and played garbage. Updated `yinsh_ml/training/augmentation.py` and `yinsh_ml/tests/test_heuristic_mcts_performance.py` to read `total_moves` from the encoder instead of hardcoding 7395. 15 tests in `yinsh_ml/tests/test_move_encoder.py` cover: layout invariants (ranges disjoint/contiguous, 85×84 movement space, network-reads-from-encoder), collision-freeness (all 7140 pairs distinct, every slot in the range reachable), self-move rejection, round-trip integrity for all four move types (placement/movement/removal enumerated exhaustively), boundary slots at sub-range edges, and the load-model guard's presence. 194 adjacent tests (all recent features + augmentation) still pass; the 14 tests in `test_heuristic_mcts_performance.py` re-green after the hardcoded-7395 fix. Per-epoch diagnostics wired into the existing `_epoch_value_diagnostics` dict so the training log now distinguishes "loss dropping but discrimination flat" from real progress. Three additions: (1) **Policy entropy** — computed per-batch as `H = -Σ p·log(p)` using the already-available `log_probs` (no recomputation). Epoch average goes into `stats_accum['policy_entropy']` via `self.last_policy_entropy` (which was referenced but never assigned — pre-existing dead code this task fixes). Rolling 20-epoch history with a 3-epoch moving mean; a WARNING fires if the current epoch drops below 50% of the recent mean, guarded by `recent_mean > 0.1` so near-zero jitter doesn't spam. (2) **Expected Calibration Error (ECE)** — 10-bin accumulator in the value-head CE block. Per batch: `value_probs = softmax(value_logits_fp32)`, confidence = max prob, prediction = argmax; bin index = `floor(confidence · 10)`. Running sums of correctness / confidence / count per bin, ECE computed at epoch end as `Σ_b (n_b / N) · |acc_b − conf_b|`. (3) **Reliability sparkline** — compact ASCII rendering of per-bin `acc − conf` gaps using 8 block glyphs (`▁▂▃▄▅▆▇█`); empty bins render as spaces. Well-calibrated bins center (`▄`), overconfident bins go low (`▁`), underconfident high (`█`) — the visual shape of the sparkline tells the story at a glance. Log line looks like: `[Value Diagnostics] ECE=0.0421, reliability=  ▄▅▄▃▂▁▁  bin_counts=[...]`. 15 tests in `yinsh_ml/tests/test_value_head_diagnostics.py` cover: perfect calibration → ECE=0, systematic overconfidence → ECE=0.4, worst-case bound, empty-diag None-return, missing-keys None-return (resume-from-older-ckpt safety), sparkline length = bin count, empty bins → spaces, well-calibrated → middle glyph, extreme overconfidence → leftmost, extreme underconfidence → rightmost, entropy-attr wiring, no-warning-on-stable-history, warning-on-sudden-drop, guard-below-0.1-mean, history-length-bounded. 127 adjacent tests still pass. Implementation note: ECE binning runs on `value_logits_fp32` (the explicit fp32 cast from the autocast path) rather than the raw autocast bf16 logits — calibration's whole sensitivity is in the probability tails that bf16 would lose. Ties directly into the §5 discrimination-plateau diagnosis: if ECE stays high even as loss drops, the plateau is calibration-architectural; if ECE shrinks, it's training-signal.

- [x] **Mixed-precision training on MPS / CUDA (Track A #1).** Wrapped the two forward passes + their loss computations in `torch.autocast(device_type=..., dtype=torch.bfloat16)`. bf16 chosen explicitly (not the PyTorch-MPS fp16 default) because bf16 shares fp32's dynamic range — gradients don't underflow, no `GradScaler` needed. Value-head CE path does an explicit `value_logits.float()` cast before `log_softmax` / `cross_entropy` — PyTorch's autocast promotion list is *per-device* (CUDA promotes `log_softmax`, MPS/CPU don't), so pinning the cast at the boundary guarantees fp32 value-head gradients regardless of runtime hardware. New trainer helper `_autocast()` returns a fresh context per `with` block (autocast contexts can't be reused). Trainer init probes support by constructing + discarding a bf16 autocast context; on failure (custom wheels, WASM targets, unforeseen platforms) it logs a warning pointing at the ≥2.7 recommendation, flips `_autocast_enabled=False`, and records the reason. When disabled, `_autocast()` returns `contextlib.nullcontext()` — crucial on older PyTorch builds where even `torch.autocast(device_type='mps', enabled=False)` raises at construction because device_type is validated before the enabled check. That deferred-crash pattern (probe succeeds at init, then `_autocast()` raises during the first `train_step` after self-play has already run) was the actual bug discovered during implementation; fixed by the nullcontext fallback and pinned by two regression tests. Plumbed through `TrainingSupervisor` → `scripts/run_training.py` → `configs/training.yaml` (`trainer.enable_autocast: true`) + smoke. 9 tests in `yinsh_ml/tests/test_autocast.py` cover: bf16 conv output, `cross_entropy` fp32 promotion, `enabled=False` no-op, explicit-cast safety pattern, CPU auto-disable, unsupported-device graceful fallback (simulates via monkeypatch), nullcontext-when-disabled invariant, signature + helper wiring.
