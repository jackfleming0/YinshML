The System Map

Subsystem boundaries and the objects that cross them.

The four zones

YinshML is an AlphaZero-style loop. Strip away the telemetry and memory pooling and the dependency graph collapses into four zones: a Game Core that owns the only mutable truth (GameState), a Search zone that turns one position into a move distribution, a Learning zone that turns positions into tensors and tensors into weights, and an Orchestration zone that runs the whole self-play→train→eval cycle. Every arrow in the diagram is a concrete Python object; the seams are where those objects change representation (GameState → tensor → logits → visit counts → weights).

Click any component below to see the key object it produces or consumes and the file it lives in. The two entrypoints at the top — TrainingSupervisor.train_iteration() and run_large_scale_selfplay.py — are the only things you ever invoke directly; everything else is wired beneath them.

Package layout — who owns what

Each package under yinsh_ml/ is the source of truth for exactly one object. When you need to change how that object looks, this is the directory you edit — and the seam you have to re-validate downstream.

yinsh_ml/ The framework package
game/ Owns GameState, Board, Move, Player/GamePhase enums, and ZobristHasher. The rules engine.
utils/ Owns StateEncoder (encoding.py) + EnhancedStateEncoder. GameState ↔ np.float32 tensor; defines total_moves = 7433. Also tournament + elo_manager + metrics_logger.
network/ Owns YinshNetwork (model.py) + NetworkWrapper (wrapper.py). Tensor -> (move_probs[7433], value). Hard-fails on policy/channel mismatch.
training/ Owns MCTS (self_play.py), YinshTrainer (trainer.py), TrainingSupervisor (supervisor.py). The search engine, the optimizer loop, and the orchestrator.
search/ Owns TranspositionTable + NodeType enums. Position cache (Zobrist-keyed, depth-preferred). NB: search/mcts.py was excised 2026-05-21 (dead duplicate).
heuristics/ Owns YinshHeuristics evaluator + the 7 learned, phase-weighted features + phase detection. GameState -> scalar value.
agents/ Owns HeuristicAgent: pure negamax + alpha-beta over the heuristic evaluator, with built-in transposition table.
self_play/ Owns SelfPlayRunner (game_runner.py), Policy classes (policies.py), GameRecord, and ParquetDataStorage (data_storage.py). The data-only generation + persistence path.
memory/ Owns GameStatePool + TensorPool + adaptive sizing. Object reuse to cut GC pressure; injected into SelfPlay/MCTS by the supervisor.
viz/ Owns the live game viewer: board_render.py (hex Figure) + game_replay.py (Parquet -> reconstructed Board/GameState).

The objects that cross the seams

Six principal objects move between subsystems. Each is tagged producer → consumer; the shape is load-bearing, so the exact type is given. Section 02 dissects each of these in detail.

GameState — Game Engine → everyone

The mutable truth: board, phase, current_player, move history, scores. Read by encoder, MCTS, heuristics, and the runner; mutated only inside yinsh_ml/game/.

▲

⚲

Normalized root visit distribution over 7433 slots from MCTS.search_batch() (yinsh_ml/training/self_play.py). This — not the raw network policy — is the cross-entropy target.

▦

(state, policy, value, metadata) tuples. Streamed to Parquet by SelfPlayDataManager (data path) or fed straight into YinshTrainer's replay buffer (in-loop path).

⚒
updated weights — Trainer → NetworkWrapper

YinshTrainer mutates the shared NetworkWrapper in place, so the next self-play iteration plays with the freshest weights (the supervisor re-points self.self_play.network each iteration).

The two entrypoints

Everything above is reachable from exactly two top-level drivers. Know which one you are in before you touch anything: one closes the learning loop, the other only opens the data spigot.

⚙
Full loop — TrainingSupervisor.train_iteration()

One self-play → train → eval → select cycle (yinsh_ml/training/supervisor.py:1003). It re-points self.self_play.network at the current weights, applies the heuristic-weight curriculum and iteration-aware Dirichlet taper, generates num_games in batches of 10, folds each game's experiences into the trainer's replay buffer, trains, then runs the tournament + anchor eval and updates best_model.pt. Weights are never reverted by default (AlphaZero-style continuous training); the gate is opt-in via revert_self_play_on_gate_failure.

▶
Data-only path — run_large_scale_selfplay.py

Drives SelfPlayRunner (yinsh_ml/self_play/game_runner.py) with a chosen policy to emit GameRecords into Parquet. No network training, no weight updates, no tournament — this is the corpus generator behind the 100K-game analysis, not the learning loop. Use it when you want data, not a better model.

The encoded state tensor & network I/O

This is the seam where game logic becomes math. encode_state turns a GameState into a fixed (6,11,11) tensor; the network maps it to a policy vector and a scalar value.

CODE 
def encode_state(self, game_state) -> np.ndarray:
    state = np.zeros((6, 11, 11), dtype=np.float32)
    # 0: current rings   1: opponent rings
    # 2: current markers 3: opponent markers
    # 4: valid-move sources mask
    phase_value = game_state.phase.value / (len(GamePhase)-1)
    state[5] = phase_value          # ch 5 broadcast
    state[5, 0, 0] = sentinel   # A1: side-to-move
    return state
PLAIN ENGLISH

Shape (6,11,11) float32 (encoding.py:330). 6 channels over the 11×11 grid; only 85 cells are real, the rest are structural padding.

Channels 0–3 are side-normalized: index 0/2 are always the current player's rings/markers, 1/3 the opponent's — not white/black. The network never sees raw color.

Channel 4 is the valid-move source mask; channel 5 broadcasts the phase value, then overwrites off-board cell A1 with a 0/1 side-to-move sentinel.

Enhanced (15-ch): EnhancedStateEncoder (enhanced_encoding.py:55) keeps 0–3, adds row threats (4–5), partial rows (6–7), ring mobility (8), center distance (9), ring influence (10), valid dests (11), phase (12), turn number (13), score diff (14).

⚠
Shape invariant: side recovery lives in channel 5, not color

Because channels 0–3 are "current player first," decode_state cannot tell white from black by channel. It reads side-to-move from the A1 sentinel (row_idx=0, col_idx=0) on channel 5, and the phase from on-board cell F6 (row 5, col 5). A1 is off the hex board (column A only has rows 2–5), so the sentinel never collides with real state and survives D2 augmentation. In the 15-ch encoder the phase/sentinel move to channel CH_GAME_PHASE = 12 — magic-indexing channel 5 there reads opponent row-threats and silently breaks.

CODE 
def predict(self, state_tensor, move_mask=None,
            temperature=1.0) -> Tuple[Tensor, Tensor]:
    # state_tensor: (6, 11, 11)
    move_logits, value = self.network(state_tensor)
    if move_mask is not None:
        move_logits = move_logits.masked_fill(~move_mask, -inf)
    move_probabilities = F.softmax(move_logits/temperature)
    return move_probabilities, value
PLAIN ENGLISH

Input: a (6,11,11) tensor (wrapper.py:273). forward() (model.py:377) runs conv trunk → two heads.

Policy head (model.py:148): Linear(..., total_moves) → raw logits of length 7433. predict masks illegal slots to -inf, then softmaxes to probabilities.

Value head: a scalar — (batch,) in classification mode, (batch,1) in regression. Already tanh'd inside the model; do not tanh again.

Produced by the network; consumed by MCTS as priors + leaf value. load_model hard-fails on any policy-size mismatch — no silent slot filtering.

⚠
Shape invariant: policy length MUST equal total_moves

Every policy vector — network output, MCTS target, stored Parquet row — is length 7433. The slot layout is contiguous and collision-free: placement [0,85), ring movement [85,7225) (src·84 + adjusted_dst), ring removal [7225,7310), marker removal [7310,7433) (123 enumerated 5-in-a-row lines). Changing _enumerate_remove_markers_lines() reshuffles slots and breaks every saved checkpoint; NetworkWrapper.load_model hard-fails on size mismatch rather than silently filtering. The pre-fix legacy size was 8390 — not load-compatible.

These types are produced and consumed at one critical seam: the GameState ⇄ Network boundary, where encode_state serializes a position and predict returns policy + value. Section 3 walks that seam end to end — how a live GameState becomes a tensor, crosses into the network, and returns as a move distribution the search layer can act on.

How a GameState becomes a tensor becomes a policy + value becomes a Move.

Three objects meet at this seam: a GameState, the StateEncoder, and the NetworkWrapper (which itself owns a YinshNetwork). Encode: GameState → (6,11,11)

encoding.py:323. The encoder is side-normalized: channels 0/2 are always the current player's rings/markers, 1/3 the opponent's. Channel 4 is the valid-moves mask written from each move's source cell; channel 5 broadcasts the phase and overrides the off-board A1 cell with a side-to-move sentinel.

CODE 
def encode_state(self, game_state: GameState) -> np.ndarray:
    state = np.zeros((6, 11, 11), dtype=np.float32)
    is_white = (game_state.current_player == Player.WHITE)
    # ch 0/1 rings, ch 2/3 markers: current-first
    state[0, r, c] = 1.0   # current rings
    # ch 4: valid-move sources
    for move in game_state.get_valid_moves():
        state[4, src_row, src_col] = 1.0
    # ch 5: phase broadcast + A1 sentinel
    state[5] = phase_value
    state[5, 0, 0] = WHITE_SENTINEL if is_white else BLACK_SENTINEL
    return state
PLAIN ENGLISH

Returns a dense numpy array, not a torch tensor. The wrapper does the from_numpy().float() conversion and device move.

Side-to-move decides which color lands in the "current" channels, so the network never sees raw white-vs-black.

Channels 0-3 are the four piece planes. Coordinates are (row-1, col-'A'), clamped to the 11x11 grid.

Channel 4 is the move-source mask: one hot cell per ring that can legally move this turn. This is the only "valid moves" signal baked into the tensor.

Channel 5 carries phase everywhere, then stamps the off-board A1 cell with a 0/1 sentinel so decode_state can recover side-to-move without flipping colors.

Predict: tensor → (move_probs, value)

wrapper.py:273. The single-position path. It runs self.network(state_tensor), masks invalid slots with -inf before softmax, scales by temperature, and returns probabilities plus the value. The value is not re-tanh'd here — the network already bounds it (model.py:139 region / classification expectation), so a second tanh would over-compress.

CODE 
def predict(self, state_tensor,
            move_mask=None, temperature=1.0):
    self.network.eval()
    with torch.no_grad():
        move_logits, value = self.network(state_tensor)
        # value already tanh'd by net - NO re-tanh
        if move_mask is not None:
            move_logits = move_logits.masked_fill(
                ~move_mask, float('-inf'))
        scaled = move_logits / temperature
        probs = F.softmax(scaled, dim=1)
        return probs, value
PLAIN ENGLISH

Args: a (6,11,11) (or batched) tensor, an optional boolean mask, and a temperature.

Eval mode + no-grad: this is pure inference, no autograd graph built.

One forward call yields raw policy logits of length 7433 and the value head output.

The commented-out torch.tanh(value) is the load-bearing absence: the value is bounded once, inside the net.

Masking sets illegal slots to -inf so softmax drives them to exactly zero probability.

Returns a normalized distribution over all 7433 slots and the scalar value.

The batched path MCTS actually calls

wrapper.py:617. predict_batch takes a list of GameStates, encodes each into one pooled batch tensor, runs a single forward pass, and returns raw logits (not softmaxed, not masked). This is deliberate: the caller — batched MCTS — applies its own masking and softmax per leaf. Temperature is folded into the logits when it is not 1.0.

CODE 
def predict_batch(self, game_states, temperature=1.0):
    batch_tensor = self._acquire_input_tensor(len(game_states))
    for i, gs in enumerate(game_states):
        arr = self.state_encoder.encode_state(gs)
        batch_tensor[i] = torch.from_numpy(arr).float()
    with torch.no_grad():
        policy_logits, values = self.network(batch_tensor)
        if temperature != 1.0:
            policy_logits = policy_logits / temperature
        return policy_logits, values  # raw, caller masks
PLAIN ENGLISH

Args: a list of GameState objects (one per MCTS leaf) and a temperature.

The batch input tensor comes from the TensorPool, sized (batch, 6, 11, 11), and is released in a finally block.

Each state is encoded and copied into its batch row — same encode_state as the singleton path.

One forward pass over the whole batch is the throughput win over per-leaf calls.

Returns (batch_size, 7433) logits and (batch_size, 1) values, both unmasked — masking and softmax are the search layer's job.

Inside the net: two heads, one trunk

model.py:377. forward runs the shared conv trunk, then splits. The policy head is Conv2d(c,64,1) → Flatten → Linear(64·11·11, 1024) → Linear(1024, total_moves) emitting raw logits (model.py:148). In classification value mode the value head's logits are softmaxed and dotted with outcome_values to yield an expectation already in [-1, 1] — that is the single tanh-equivalent bounding referenced by predict.

CODE 
def forward(self, x) -> Tuple[Tensor, Tensor]:
    x = self.conv_block(x)            # shared trunk
    for block in self.main_blocks: x = block(x)
    policy = self.policy_head(x)     # (B, 7433) logits
    value_output = self.value_head(x)
    if self.value_mode == 'classification':
        probs = F.softmax(value_output, dim=-1)
        value = (probs * self.outcome_values).sum(-1)
        self._value_logits = value_output  # for CE loss
    else:
        value = value_output.squeeze(-1)  # tanh'd scalar
    return policy, value
PLAIN ENGLISH

Args: (batch, input_channels, 11, 11) — 6 channels basic, 15 enhanced.

Trunk: an entry conv block then a stack of residual/attention blocks, shared by both heads.

Policy head ends in a bare Linear(..., total_moves) — raw logits, no activation, so masking and temperature happen downstream.

Classification value mode turns class logits into a probability-weighted expectation over the discrete outcome centers; the result is already in [-1,1].

The raw class logits are stashed on _value_logits so the trainer can run cross-entropy without changing this interface.

Returns (policy_logits, value) — the exact pair every predict path consumes.

Hard contracts at the seam

⚠
Policy length and channel count are load-bearing invariants

The policy head length must equal StateEncoder().total_moves == 7433 (85 + 7140 + 85 + 123, encoding.py:146). This number is the move-index layout: slot ↔ move meaning is fixed by it. NetworkWrapper.load_model (wrapper.py:423) hard-fails on three mismatches rather than silently filtering — input-channel count (wrapper.py:434), value-head mode/classes (wrapper.py:450), and policy-head size (wrapper.py:492). Silent filtering would leave the offending layer randomly initialized and play garbage. Known past policy sizes: 7395 (pre-rework), 8390 (collision-prone REMOVE_MARKERS), 7433 (current). Switching sizes means retrain or explicit migration — old checkpoints will not load.

The use_enhanced_encoding plumbing

A single boolean threads channel count from the wrapper constructor down to both the encoder choice and the network's input conv. load_model can also flip this flag to match a checkpoint's conv-in dimension (wrapper.py:78), so the encoder instantiated at wrapper.py:200 always agrees with the loaded weights.

1
NetworkWrapper(use_enhanced_encoding=..., num_channels=..., num_blocks=..., value_mode='classification', num_value_classes=7, value_head_type=...)

wrapper.py:26. The flag sets input_channels = 15 if use_enhanced_encoding else 6. When a model_path is given, channel count, block count, value mode, and head type are auto-detected from the state_dict and can override the flag.

2

self.use_enhanced_encoding (post-autodetect) picks EnhancedStateEncoder() (15-ch) or StateEncoder() (6-ch). Reading the post-detect flag, not the constructor arg, is what prevents an encoder/weights mismatch that would yield 0-move games.

3
YinshNetwork(input_channels=6 or 15, ...)

wrapper.py:161 → model.py:82. The same channel count sizes the entry Conv2d(input_channels, num_channels, 3, padding=1). Both sides of the seam read one source of truth, so the conv-in dimension and the encoder's output channels can never silently drift.

total_moves = 7433 85 ring placements + 7140 ring moves + 85 marker removals + 123 REMOVE_MARKERS lines (encoding.py:146).
input_channels = 6 | 15 Basic vs enhanced encoder; the only thing that changes the trunk's first conv.

Downstream, predict_batch is the hot path: batched MCTS collects many leaf GameStates, hands them to this seam in one shot, and applies its own masking and priors to the returned logits. That consumer — where evaluation meets search — is Section 4: the Search ↔ Evaluation seam.

How evaluation_mode picks the leaf scorer

The same string switch governs both leaf paths (_evaluate_state at self_play.py:1307 for the singleton path, _evaluate_and_backup_batch at 1111 for the batch). Three branches:

N

pure_neural

Leaf value and priors come entirely from the network. predict_batch logits are softmaxed into priors; the value head supplies the backed-up value. The heuristic evaluator is never touched.

H

pure_heuristic

YinshHeuristics.evaluate_position scores each leaf (states evaluated individually — heuristics do not batch). Priors are a uniform distribution over legal moves.

⚖

hybrid

Both run. With w_h = heuristic_weight, w_n = 1 - w_h: value = w_n·value_nn + w_h·value_heuristic and the priors blend the same way (self_play.py:1145–1160).

⚠
Offense-only equilibrium risk

All 7 heuristic features are attack-oriented and differential. Defense is only captured implicitly: it shows up as the opponent's term in the differential, which negamax can only act on at search depth ≥3. At depth 1–2 (shallow rollouts, pure-heuristic mode) the engine can drift into one-sided attacking play with no defensive response. The Trajectory tab of the live viewer exists to spot this signature — sustained one-sided feature growth without reply.

The Self-Play ↔ Storage ↔ Training Seam

From a finished game to a gradient step, and the supervisor that drives it.

Every training iteration is one turn of the same crank: TrainingSupervisor.train_iteration() (yinsh_ml/training/supervisor.py:1003) generates self-play games, folds the resulting (state, policy, value) tuples into the replay buffer, trains the network for N epochs, then runs a tournament and either promotes or discards the new weights. The seam has two distinct sides — an in-memory replay buffer (the AlphaZero training path the supervisor actually drives) and a Parquet archive (the analysis/audit path through game_runner.py). Producer: how a game becomes experiences

The training-path producer is the per-game worker in yinsh_ml/training/self_play.py (not game_runner.py, which feeds the Parquet archive). At each learner position it appends three parallel lists. The policy is the MCTS visit-count distribution over all total_moves slots; the value is the final game outcome, signed for the side that was to move.

CODE 
# self_play.py:2095 — store target only on the learner's turn
if store_target and is_learner_turn:
    encoded_state = state_encoder.encode_state(state).astype(np.float32)
    states.append(encoded_state)
    policies.append(move_probs)      # MCTS visit dist
    players.append(state.current_player)

# self_play.py:2142 — outcome from final score, clipped to [-1, 1]
score_diff   = state.white_score - state.black_score

values = [outcome_white if p == Player.WHITE
         else -outcome_white for p in players]
PLAIN ENGLISH

Only record a sample when it is the learner's turn (cross-teacher games skip the opponent's positions).

Encode the board to its 6- or 15-channel tensor — the network input.

The policy target is move_probs, MCTS's normalized visit counts over every move slot. Visited-more = trained-toward-more.

Remember whose turn it was so the value sign can be assigned later.

The whole game shares one outcome: white's score minus black's, divided by 3 (a win is 3 rows) and clamped to [-1, 1].

Flip the sign for black-to-move positions, so every value is "did the side to move end up winning?"

The terminal position is appended with a zero policy (self_play.py:2130) so it trains only the value head. These three lists flow straight into trainer.add_game_experience(states, policies, values, …) (trainer.py:116) — note the buffer stores the position-specific MCTS root value as the value target, not a re-derived margin.

⚠
Long games are subsampled before storage

In GameExperience.add_game_experience() at trainer.py:152, any game with len(states) > 100 is thinned: keep the first 20 and last 20 positions, then randomly sample up to 30 of the middle ones (trainer.py:154-172). States, policies, values, and move-numbers are all sliced by the same keep_indices, so they stay aligned. This caps redundancy from long, low-information midgames before they bloat the buffer. Toggle with the subsample_long_games constructor flag.

Storage: replay buffer vs. Parquet archive

The training path keeps experiences in RAM in GameExperience (parallel lists states / move_probs / values / phases / move_numbers, trainer.py); policies are downcast to float16 to halve buffer memory (trainer.py:206). The analysis/audit path serializes full games to Parquet through ParquetDataStorage (self_play/data_storage.py:50), batched one file per StorageConfig.batch_size games (default 100).

💾
Parquet vs. JSON: ~500× smaller

Columnar Snappy-compressed Parquet stores a game in roughly ~3 KB versus ~1.5 MB for the equivalent JSON — about a 500× reduction, plus fast parallel loads via PyArrow. The 100K-game corpus is ~344 MB of Parquet. Always store training/analysis data as Parquet; JSON is the degraded fallback only.

Consumer: the loss the trainer actually computes

YinshTrainer samples mixed batches and computes two losses with two optimizers (the policy head and the value-head-plus-trunk are stepped separately so the trunk gets both signals). The policy loss is a cross-entropy against the MCTS visit distribution.

CODE

PLAIN ENGLISH

Temperature-scale the policy logits, then log-softmax across all 7433 move slots.

Cross-entropy: -(target * log_pred) summed over moves, averaged over the batch. target_probs is the MCTS visit distribution; only valid moves carry mass.

Value head is mode-switched. In 'classification' mode the scalar outcome is bucketed into 7 ordinal classes (score-diff -3..+3) and trained with cross-entropy.

In legacy 'regression' mode the value head instead predicts a scalar directly and is trained with MSE plus a variance penalty that discourages collapse.

The reported total is just the sum; in practice each head is back-propped through its own optimizer (see trainer.py:1115-1331).

value_mode lives on the network (YinshNetwork) and is read at trainer.py:1170; classification additionally maps the soft target with an optional Gaussian spread (soft_value_target_sigma, trainer.py:1189) to encode ordinal structure. Epoch-level accumulation and the entropy-collapse watch run in train_epoch.

Cross-Cutting Infrastructure & Invariants

Shared plumbing, and the contracts that break the build if you touch one side.

Shared memory infrastructure

Three packages under yinsh_ml/memory/ exist purely to reduce allocation churn during self-play and training. None of them own behavior — they are injected dependencies: the consumer accepts a pool through a constructor argument and threads it down. Know the argument name and you know how to wire (or unwire) it.

🗃️
GameStatePool — yinsh_ml/memory/game_state_pool.py:199

Reuses GameState objects to cut GC pressure during deep search. Threaded into the MCTS engine via the game_state_pool= arg (self_play.py:171 and :1504); when present, get() / return_game_state() replace fresh allocations. Also passed by the self-play game runner. Bypassed entirely when self_play.use_cpp_engine: true — the C++ State.clone() beats any pool reuse (scripts/run_training.py:317).

🧮
TensorPool — yinsh_ml/memory/tensor_pool.py:114

Reuses inference tensors for batched NN evaluation. Passed to the network via NetworkWrapper(tensor_pool=...) (wrapper.py:27); the wrapper draws policy/value buffers of shape (batch, _policy_size) from it instead of calling torch.zeros each batch (wrapper.py:239). Configured via TensorPoolConfig.

📈
AdaptivePoolSizer — yinsh_ml/memory/adaptive.py:197

Resizes pools at runtime from hit-rate/utilization windows (create_adaptive_pool_sizer(), adaptive.py:372). Wraps a pool rather than replacing it — the consumer still sees a plain GameStatePool/TensorPool.

⚡
ZeroCopyTensorFactory — yinsh_ml/memory/zero_copy.py:81

Builds pinned/shared-storage tensors for zero-copy host→GPU transfer, plus InPlaceOperations (zero_copy.py:255) for buffer-reusing math. A throughput optimization on the batched-inference path; not on the correctness path.

🔌
Pools are optional and default-None

Every consumer treats its pool arg as Optional[...] = None and guards with a _pool_enabled = pool is not None flag (self_play.py:278). Code paths must stay correct with the pool absent — the pool only changes where memory comes from, never what the result is. A pool that returns a stale, un-reset object is a silent correctness bug, not a performance one.

Change-one-change-all invariants

These are the highest-risk edits in the codebase: a single constant or convention duplicated across multiple files, where changing one side silently (or loudly) corrupts the other. Each card names the invariant, the files on both sides, and the failure mode. The canonical source for most of these is the Foundational Rule Fixes section of CLAUDE.md.

1
Policy size = 7433 = StateEncoder().total_moves

total_moves = 85 + 7140 + 85 + 123 = 7433 (utils/encoding.py:146) defines the policy-head width in YinshNetwork (network/model.py). On load, NetworkWrapper.load_model compares the checkpoint's policy_head.7.weight rows against the live encoder and hard-fails on mismatch (wrapper.py:473–503). Failure mode: old 8390-slot checkpoints (pre-April-2026 REMOVE_MARKERS fix) are not load-compatible — a clear RuntimeError, not a silent wrong-move map.

2
Channel count: 6 (basic) vs 15 (enhanced) must agree across encoder ↔ model ↔ checkpoint

StateEncoder (6-ch, encoding.py) or EnhancedStateEncoder (15-ch, enhanced_encoding.py) must match YinshNetwork(input_channels=...) and the saved conv_block.0.weight dim-1. The wrapper auto-detects channels from the checkpoint (wrapper.py:82–88) and hard-fails on encoder mismatch (wrapper.py:428–443). Failure mode: switching channel counts requires regenerating supervised data + retraining — there is no silent channel filtering.

3
Zobrist hash MUST include board + side-to-move + game phase

ZobristHasher.hash_state XORs in a side_to_move_key and a per-phase key (game/zobrist.py:114–119,178–185), with O(1) toggle_side_to_move / toggle_phase helpers. Failure mode: drop either component and the transposition table (search/transposition_table.py) returns wrong-position hits — identical piece layouts with a different mover or phase collide and the cached eval is silently wrong.

4
Channels 0–3 are side-normalized; recover the mover from the A1 sentinel on channel 5

The encoder writes pieces as “current player first” (channels 0–3), then stamps a 0/1 side-to-move sentinel at off-board cell A1 (row=0,col=0) on channel 5 (encoding.py:384–391). decode_state reads the mover from that sentinel (encoding.py:589–596). Failure mode: any code that re-flips colors by current_player double-inverts the board; A1 is off the hex grid (column A only spans rows 2–5) so it never collides with real state and survives D2 augmentation.

5
DIRECTIONS / HEX_LINE_AXES are the single source of truth for hex lines

constants.py:96 (DIRECTIONS, 3 forward axes) and constants.py:127 (HEX_LINE_AXES, 6 signed) define every legal line. The opposite-sign pseudo-diagonal (1, -1) / (-1, 1) is NOT a board line (constants.py:84–87). Failure mode: reintroducing it (in find_marker_rows / is_valid_marker_sequence) revives the pseudo-diagonal row bug — phantom rows, illegal captures, and a policy-space size drift.

6
value_mode + num_value_classes must agree across wrapper, value head, and trainer loss

NetworkWrapper(value_mode='classification'|'regression', num_value_classes=7) (wrapper.py:28) selects the value-head shape and the trainer's loss (cross-entropy vs MSE). load_model hard-fails on a mode mismatch (wrapper.py:445–458) and on a class-count mismatch (wrapper.py:465–472). Failure mode: a regression checkpoint loaded into a classification wrapper (or a class-count drift) — rejected with a RuntimeError telling you to re-instantiate with the matching mode.

⚠️
All six are guarded, but only some loudly

Invariants 1, 2, and 6 throw at load_model time — you find out immediately. Invariants 3, 4, and 5 fail silently: the program runs, produces plausible numbers, and quietly trains on a corrupted target. Treat any edit to zobrist.py, the encoder sentinel, or constants.py hex axes as a full-test-suite change.
