The Search ↔ Evaluation Seam

What MCTS is wired to, what it returns, and how leaves get scored.

There is exactly one MCTS engine in the codebase: yinsh_ml/training/self_play.py::MCTS. Its constructor declares every collaborator the search leans on — the network it queries at leaves, an optional heuristic evaluator, a memory pool, and an optional shared inference evaluator that coalesces GPU calls across instances. Read the signature as the seam's contract.

search() (self_play.py:616) — consumes a GameState + the outer-game move number (for temperature / Dirichlet tapering). Runs the budget of simulations, scoring one leaf per sim via _evaluate_state.

Returns a length-total_moves vector of root child visit counts (the improved policy that self-play samples its move from).

search_batch() (self_play.py:839) — same contract, but collects up to batch_size leaves before evaluating them in one network call. This is the path used in training and bulk evaluation; search() is the singleton fallback.

move_number drives the rollout budget (early vs late sims) and root Dirichlet-noise mixing — not just temperature.

The batched leaf-evaluation loop

search_batch descends the tree per simulation, but instead of scoring each leaf inline it queues the leaf's GameState. When the queue fills (or the budget ends) it flushes through _evaluate_and_backup_batch (self_play.py:1089), which calls predict_batch once for the whole batch — or hands off to the shared evaluator if one was injected. The shared evaluator and predict_batch return the same shape, so the only thing that changes is who runs the GPU call. Actor 2 is skipped entirely when evaluator is None.

How evaluation_mode picks the leaf scorer

The same string switch governs both leaf paths (_evaluate_state at self_play.py:1307 for the singleton path, _evaluate_and_backup_batch at 1111 for the batch). Three branches:

N

pure_neural

Leaf value and priors come entirely from the network. predict_batch logits are softmaxed into priors; the value head supplies the backed-up value. The heuristic evaluator is never touched.

H

pure_heuristic

YinshHeuristics.evaluate_position scores each leaf (states evaluated individually — heuristics do not batch). Priors are a uniform distribution over legal moves.

⚖

hybrid

Both run. With w_h = heuristic_weight, w_n = 1 - w_h: value = w_n·value_nn + w_h·value_heuristic and the priors blend the same way (self_play.py:1145–1160).

When the heuristic branch is live, it sits on top of a small stack — each layer feeds the one above it:

🎯
YinshHeuristics — heuristics/evaluator.py

evaluate_position(state, player) returns a differential scalar (positive = good for that player). The seam's entry point into the heuristic stack.

🧮
features.py — the 7 features

extract_all_features(state, player) (features.py:495) computes completed-runs differential, ring centrality, ring spread, potential runs, connected chains, ring mobility, edge proximity.

⚖
weight_manager.py — phase-specific weights

WeightManager (weight_manager.py:15) holds per-phase weight dicts keyed early / mid / late; features are weighted by the active phase.

🕒
phase_detection.py — classifies the phase

detect_phase / get_phase_weights map a GameState to its phase bucket so the weight manager knows which column to read.

⚠
Offense-only equilibrium risk

All 7 heuristic features are attack-oriented and differential. Defense is only captured implicitly: it shows up as the opponent's term in the differential, which negamax can only act on at search depth ≥3. At depth 1–2 (shallow rollouts, pure-heuristic mode) the engine can drift into one-sided attacking play with no defensive response. The Trajectory tab of the live viewer exists to spot this signature — sustained one-sided feature growth without reply.

Shared search infrastructure

The heuristic engine has a sibling: HeuristicAgent (agents/heuristic_agent.py), a pure negamax + alpha-beta player. It does not reuse the MCTS engine, but it leans on the same position-caching primitives — the transposition table and Zobrist hasher — so improvements there pay off in both search paths.

TranspositionTable search/transposition_table.py — Zobrist-keyed position cache. Depth-preferred replacement (keeps the deeper search), default size_power=20 (2^20 = ~1M entries). Stores (value, best_move, depth, node_type) per store().
ZobristHasher.hash_state game/zobrist.py:359 — hashes board and side-to-move and game phase, so two identical layouts differing only in current_player or phase hash apart. The TT key.
toggle_side_to_move game/zobrist.py:372 — O(1) incremental hash flip when only the side-to-move changes; avoids a full re-hash of the board.
toggle_phase game/zobrist.py:392 — O(1) incremental flip when the game phase transitions (e.g. RING_PLACEMENT → MAIN_GAME).
HeuristicAgent._negamax agents/heuristic_agent.py:316 — negamax + alpha-beta, probing the same TT via _zobrist_hasher.hash_state (heuristic_agent.py:327–328) and storing EXACT / LOWER_BOUND / UPPER_BOUND entries by the alpha-beta window.
use_transposition_table agents/heuristic_agent.py:70 — on by default in HeuristicAgentConfig; transposition_table_size_power (default 20) and zobrist_seed are configurable alongside it.
