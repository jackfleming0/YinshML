The Self-Play ↔ Storage ↔ Training Seam

From a finished game to a gradient step, and the supervisor that drives it.

Every training iteration is one turn of the same crank: TrainingSupervisor.train_iteration() (yinsh_ml/training/supervisor.py:1003) generates self-play games, folds the resulting (state, policy, value) tuples into the replay buffer, trains the network for N epochs, then runs a tournament and either promotes or discards the new weights. The seam has two distinct sides — an in-memory replay buffer (the AlphaZero training path the supervisor actually drives) and a Parquet archive (the analysis/audit path through game_runner.py). Both are documented below; do not confuse them.

Producer: how a game becomes experiences

The training-path producer is the per-game worker in yinsh_ml/training/self_play.py (not game_runner.py, which feeds the Parquet archive). At each learner position it appends three parallel lists. The policy is the MCTS visit-count distribution over all total_moves slots; the value is the final game outcome, signed for the side that was to move.

CODE

if store_target and is_learner_turn:
    encoded_state = state_encoder.encode_state(state).astype(np.float32)
    states.append(encoded_state)
    policies.append(move_probs)      # MCTS visit dist
    players.append(state.current_player)

# self_play.py:2142 — outcome from final score, clipped to [-1, 1]
score_diff   = state.white_score - state.black_score
outcome_white = float(np.clip(score_diff / 3.0, -1.0, 1.0))
values = [outcome_white if p == Player.WHITE
         else -outcome_white for p in players]
PLAIN ENGLISH

Only record a sample when it is the learner's turn (cross-teacher games skip the opponent's positions).

Encode the board to its 6- or 15-channel tensor — the network input.

The policy target is move_probs, MCTS's normalized visit counts over every move slot. Visited-more = trained-toward-more.

Remember whose turn it was so the value sign can be assigned later.

The whole game shares one outcome: white's score minus black's, divided by 3 (a win is 3 rows) and clamped to [-1, 1].

Flip the sign for black-to-move positions, so every value is "did the side to move end up winning?"

The terminal position is appended with a zero policy (self_play.py:2130) so it trains only the value head. These three lists flow straight into trainer.add_game_experience(states, policies, values, …) (trainer.py:116) — note the buffer stores the position-specific MCTS root value as the value target, not a re-derived margin.

⚠
Long games are subsampled before storage

In GameExperience.add_game_experience() at trainer.py:152, any game with len(states) > 100 is thinned: keep the first 20 and last 20 positions, then randomly sample up to 30 of the middle ones (trainer.py:154-172). States, policies, values, and move-numbers are all sliced by the same keep_indices, so they stay aligned. This caps redundancy from long, low-information midgames before they bloat the buffer. Toggle with the subsample_long_games constructor flag.

Storage: replay buffer vs. Parquet archive

The training path keeps experiences in RAM in GameExperience (parallel lists states / move_probs / values / phases / move_numbers, trainer.py); policies are downcast to float16 to halve buffer memory (trainer.py:206). The analysis/audit path serializes full games to Parquet through ParquetDataStorage (self_play/data_storage.py:50), batched one file per StorageConfig.batch_size games (default 100).

The Parquet schema is turn-grained: each GameTurn carries the move, the player, and the 7-feature heuristic dict for that position.

Note this archive stores moves and features for replay/analysis — not the (state-tensor, policy, value) training triples. Those live only in the in-RAM buffer.

Games accumulate in current_batch...

...and once batch_size (default 100) games are queued, the whole batch is flattened to a DataFrame and written as one Snappy-compressed Parquet file.

_batch_to_dataframe() (data_storage.py:138) flattens game-level metadata onto every turn row and spreads turn.features into columns; flush() (data_storage.py:185) writes any partial trailing batch. If PyArrow/fastparquet are missing, the writer silently falls back to JSON (data_storage.py:105).

💾
Parquet vs. JSON: ~500× smaller

Columnar Snappy-compressed Parquet stores a game in roughly ~3 KB versus ~1.5 MB for the equivalent JSON — about a 500× reduction, plus fast parallel loads via PyArrow. The 100K-game corpus is ~344 MB of Parquet. Always store training/analysis data as Parquet; JSON is the degraded fallback only.

Consumer: the loss the trainer actually computes

YinshTrainer samples mixed batches and computes two losses with two optimizers (the policy head and the value-head-plus-trunk are stepped separately so the trunk gets both signals). The policy loss is a cross-entropy against the MCTS visit distribution.

CODE 
# trainer.py:1120 — POLICY LOSS: cross-entropy vs MCTS target
scaled_logits = pred_logits / self.temperature
log_probs     = F.log_softmax(scaled_logits, dim=1)
policy_loss   = -(target_probs * log_probs).sum(dim=1).mean()

# trainer.py:1170 — VALUE LOSS: classification mode
if self.network.network.value_mode == 'classification':
    ce_loss    = F.cross_entropy(value_logits_fp32, target_class)
    value_loss = ce_loss + discrimination_loss
# trainer.py:1288 — VALUE LOSS: legacy regression mode
else:
    value_loss_mse = F.mse_loss(pred_values.view(-1), target_values_flat)
    value_loss     = value_loss_mse + variance_penalty

# trainer.py:1352
total_loss = policy_loss + value_loss
PLAIN ENGLISH

Temperature-scale the policy logits, then log-softmax across all 7433 move slots.

Cross-entropy: -(target * log_pred) summed over moves, averaged over the batch. target_probs is the MCTS visit distribution; only valid moves carry mass.

Value head is mode-switched. In 'classification' mode the scalar outcome is bucketed into 7 ordinal classes (score-diff -3..+3) and trained with cross-entropy.

In legacy 'regression' mode the value head instead predicts a scalar directly and is trained with MSE plus a variance penalty that discourages collapse.

The reported total is just the sum; in practice each head is back-propped through its own optimizer (see trainer.py:1115-1331).

value_mode lives on the network (YinshNetwork) and is read at trainer.py:1170; classification additionally maps the soft target with an optional Gaussian spread (soft_value_target_sigma, trainer.py:1189) to encode ordinal structure. Epoch-level accumulation and the entropy-collapse watch run in train_epoch.

The four move-selection policies

Self-play move choice is pluggable via yinsh_ml/self_play/policies.py (PolicyFactory.create_policy, policies.py:948). Note that the supervisor's high-throughput training path drives the batched self_play.py::MCTS engine directly; these policy classes are the configurable SelfPlayRunner / analysis path.

R

RandomMovePolicy

Uniform baseline (policies.py:26) with a small rule_based_probability of phase-aware nudges (prefer center on placement, push toward the opponent). Used to seed exploration / sanity baselines.

H

HeuristicPolicy

Wraps HeuristicAgent negamax + alpha-beta (policies.py:305). Epsilon-greedy plus temperature softmax over scored moves; a fast mode skips search for <100 ms/move. Decays exploration over games.

M

MCTSPolicy

Neural-guided MCTS wrapper (policies.py:598) — now removed/raises: it bound the dead search/mcts.py engine. Production MCTS self-play goes through self_play.py::MCTS.search_batch.

A

AdaptivePolicy

Transitions heuristic → neural by training progress (policies.py:720): linear/exponential/step schedule over transition_steps, gated by a neural-quality check before switching.

Config plumbing: YAML key → component

scripts/run_training.py loads one YAML (load_config, run_training.py:22) and reads these keys into the supervisor / trainer:

self_play.games_per_iteration Games generated per iteration → train_iteration(num_games=…) (run_training.py:469,527). Default 50.
self_play.num_simulations MCTS sim budget per move → mcts_simulations on the supervisor (run_training.py:499). Default 96.
self_play.mcts_batch_size Leaf-eval batch size for search_batch (run_training.py:309). Default 32.
trainer.epochs_per_iteration Training epochs per iteration → train_iteration(epochs=…) (run_training.py:470,527). Default 40.
trainer.batch_size Minibatch size for buffer sampling (run_training.py:335). Default 256.
arena.promotion_threshold Wilson lower-bound win-rate the candidate must clear to promote (run_training.py:437, used by _should_promote, supervisor.py:3012). Default 0.55.
num_iterations Outer loop length (run_training.py:216,525). Default 50; overridable with -n.
anchor.mcts_simulations Sims for the fixed pure-neural anchor eval (run_training.py:461). Default 64.

The promotion decision is not a bare win-rate: _should_promote (supervisor.py:3012) computes a Wilson 95% lower bound on the head-to-head wins and promotes only if it exceeds promotion_threshold. A failed gate does not hard-revert — the loop continues on the latest weights (AlphaZero-style), optionally reloading best_model.pt for the next self-play round.

Compute profile: what is CPU-bound vs GPU-bound

The supervisor logs the split every iteration — Breakdown: SelfPlay=…s, Train=…s, … (supervisor.py:2110) — and in the AlphaZero regime self-play usually dominates the wall clock. The two phases load opposite resources: self-play is CPU/coordination-bound, the gradient update is GPU-bound, and almost every optimization in this seam exists to stop the first from starving the second.

CPU

Self-play generation

CPU-bound. Pure-Python game logic, MCTS tree ops (select/expand/backup), Zobrist hashing, transposition lookups and heuristic scoring all run on CPU and are parallelized across cores with a ProcessPoolExecutor (self_play.py:1845, num_workers). Game logic was hot enough to warrant an opt-in C++ bitboard engine (use_cpp_engine → CppGameState, self_play.py:1966,2237).

GPU

Network training

GPU-bound. Batched forward/backward on-device, bf16 torch.autocast on CUDA/MPS (trainer.py:719-726), zero-copy transfers. The CPU cost hiding here is batch assembly + dihedral symmetry augmentation (up to 12× expansion, trainer.py:54,61), which can starve the GPU if it is not overlapped with the step.

CPU

Arena / evaluation

CPU-bound, same shape as self-play: real games driven by MCTS + game logic, with batched net inference at the leaves. The promotion tournament therefore competes with self-play for the same CPU workers.

⚠
During self-play the GPU is the idle resource

Workers do run the net on CUDA when present (self_play.py:2203), but each MCTS only collects mcts_batch_size (default 32, self_play.py:1494) leaves at a time. N independent worker processes firing small inference calls “serialize on the GPU's command queue rather than coalescing into one” (BATCHED_EVALUATOR_DESIGN.md:40) — so the GPU sits underfed while CPU is the bottleneck. That is the entire reason for the shared InferenceEvaluator threaded mode: one model on the GPU, N threads play games against it, coalescing inference (self_play.py:1761). It trades CPU-side coordination for higher GPU utilization.

Which knob relieves which constraint

num_workers ↑ CPU (self-play/arena): more parallel game workers — the primary self-play throughput lever (self_play.py:1845).
use_cpp_engine CPU: swaps Python game logic for the C++ bitboard engine in workers (self_play.py:2237).
shared evaluator / mcts_batch_size GPU-feeding: coalesce small leaf batches so self-play stops starving the GPU (self_play.py:1761).
GameStatePool / TensorPool CPU: reuse allocations to cut GC/alloc pressure in the hot loop (yinsh_ml/memory/).
trainer.batch_size ↑ + autocast GPU (training): bigger matmuls + bf16 raise utilization — target >80% (CLAUDE.md:472-473).
zero-copy / pinned memory Transfer seam: keep host↔device copies from becoming the limiter (memory/zero_copy.py, CLAUDE.md:474).
🧮
The balance shifts with network size

With a small net on a large GPU (the common case here) self-play is heavily CPU-bound on Python game logic + MCTS. Scale the network up enough and leaf evaluation pushes back toward the GPU. Use the SelfPlay=/Train= breakdown (supervisor.py:2110) and live GPU utilization as the ground truth before tuning — measure which side you are on, do not guess.
