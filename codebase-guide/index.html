YinshML · Guided Tour
01

What YinshML Actually Does

A program that teaches itself to play a board game by playing against itself, millions of times, and keeping notes on what worked.

Start with one command

Before any architecture diagrams, here is the thing you will actually run:

python scripts/run_training.py Kicks off the whole learning loop: play games, learn from them, repeat.
python run_large_scale_selfplay.py Just generates game data, without the learning step. Useful for building a dataset.

That first command is the front door to this entire codebase. YinshML is an reinforcement learning framework for YINSH, an abstract strategy board game played on a hexagonal grid. The goal is to produce an agent that plays YINSH extremely well.

The twist: it learns with no human games at all. It is built in the style of AlphaZero — the agent starts knowing only the rules, plays against itself over and over, and gradually gets better by noticing which moves tend to lead to wins.

💡
The whole idea in one sentence

Imagine a player who becomes superhuman purely by playing themselves a few million times and writing down, after every game, which positions turned out well. No coach, no opening book, no human games. That feedback loop is the entire system — everything in this repo exists to make that loop fast and stable.

Two pieces of jargon do all the heavy lifting here. A policy is the network saying "here is how good each possible move looks." A value is the network saying "from this position, I think this side is winning by about this much." The whole point of training is to make both predictions sharper over time.

The six subsystems

The loop above is orchestrated across six major parts of the codebase. Each card is a chunk of the repo you will meet in later modules:

🎲

Game Engine

The rules of YINSH in code: the board, legal moves, when the game is over, who won. Everything else sits on top of this. yinsh_ml/game/

🧠

Neural Network

Looks at a board and outputs two things: a policy (which moves look good) and a value (who is winning). This is the part that actually learns. yinsh_ml/network/

🔍

Search / MCTS

Looks ahead by simulating many possible futures, guided by the network, to pick a stronger move than the network alone would. yinsh_ml/training/self_play.py

📐

Heuristics

Hand-crafted scoring rules (center control, marker chains, etc.) learned from 100K games. They give the search good instincts before the network is strong. yinsh_ml/heuristics/

♟

Self-Play

Runs games of the agent against itself and records every position as training data. The factory that produces experience. yinsh_ml/self_play/

🎛️

Training Supervisor

The conductor. Runs self-play, then training, then evaluation, then repeats — wiring all the pieces above into one loop. yinsh_ml/training/supervisor.py

The Training Supervisor is the one to anchor on. Its single method train_iteration() is one full turn of the loop, and its own docstring spells it out: "One full self-play → training → evaluation → model-selection cycle."

From command to loop, in code

Here is the actual entry point — the tail of scripts/run_training.py. Notice how little there is: it builds the supervisor, then calls one method in a loop.

CODE 
supervisor = TrainingSupervisor(
    network=network,
    mcts_simulations=int(sp.get("num_simulations", 96)),
    mode_settings=mode_settings,
)

for it in range(start_iteration, num_iterations):
    summary = supervisor.train_iteration(
        num_games=games_per_iteration,
        epochs=epochs_per_iteration,
    )
PLAIN ENGLISH

Build the conductor that runs everything, handing it the freshly built neural network.

Tell its look-ahead search how hard to think per move — here, 96 simulated futures.

Pass in all the knobs from the config file (learning rate, how many games, and so on).

Now just repeat the loop, once per iteration, for as many rounds as configured (default 50).

Each call to train_iteration is one full turn of the wheel: play games, learn, get tested. That single line is the heartbeat of the whole project.

Those knobs live in plain YAML at configs/training.yaml. For example, num_iterations: 50 sets how many times the loop runs, games_per_iteration: 50 sets how many self-play games feed each round, and promotion_threshold: 0.55 sets the bar a new network must clear in the tournament to be crowned the new best. You tune the experiment by editing that file, not the code.

📊
How "is the new one actually better?" gets answered

After training, the new network plays a head-to-head tournament against the reigning champion. Results feed an ELO rating so you can track strength across versions on one scale. The network is only kept if it wins clearly enough — that gate is what makes the loop a ratchet that mostly goes up.

That is the 10,000-foot view: one command, one loop, six subsystems, getting stronger each pass. Everything rests on the rules of the game itself — so Module 2 dives into the Game Engine: the hexagonal board, how moves and captures work, and how a YINSH position is represented in code.

02

The Board and the Rules

How a hexagonal board game becomes clean, testable Python — pieces, phases, and the one bug that hid in plain sight.

Picture the board first

Before a single line of code makes sense, you need the mental picture. YINSH is played on a hexagonal grid — not a square chessboard. The code models it as an 11×11 coordinate space, but the playing surface is a diamond cut out of that square: only 85 of the 121 cells are real. The corners are empty.

Each column (lettered A through K) holds a different number of valid rows, and those counts form the diamond shape: [4, 7, 8, 9, 10, 9, 10, 9, 8, 7, 4]. Column A only has rows 2–5; the fat middle columns reach all the way across. This is why you'll constantly see is_valid_position() guarding the code — most of the coordinate plane simply isn't on the board.

Two players, WHITE and BLACK, share the board. Each owns two kinds of pieces: rings (the things you actually move) and markers (tokens that get left behind and flip color). Hold those two ideas — rings move, markers accumulate — and the rest of the engine falls into place.

💡
Why integers for players?

Player.WHITE = 1 and Player.BLACK = -1 isn't arbitrary. Encoding the two sides as +1 / -1 means "the other player" is just multiplication by −1, and a board can be flipped to the current player's perspective by sign math instead of branching. This tiny choice ripples all the way up into how the neural network sees positions in Module 3.

Six files, one game engine

Everything about the rules lives in yinsh_ml/game/. There's no AI here yet — this is the pure, deterministic referee that the search algorithms and neural network sit on top of. Each file has a single, sharp job:

yinsh_ml/game/ The rules of YINSH, as plain Python — no machine learning
constants.py Board geometry & rule numbers: the 85 valid positions, the hex directions, 5 rings per player, 5 markers per row, 3 rows to win
types.py The vocabulary: Move, MoveType, GamePhase enums — small, frozen dataclasses everything else passes around
board.py Where the pieces actually are. Places/moves pieces, flips markers when a ring jumps them, finds completed rows
game_state.py The referee: whose turn it is, the current phase, move history, win/stalemate detection, applying a move
moves.py Move generation — given a state, enumerate every legal move a player could make right now
zobrist.py A fast fingerprint of a position (a single integer hash) so the search can recognize "seen this before" instantly

A useful way to read the dependency arrow: constants and types define the nouns and numbers, board holds the physical state, game_state is the rules engine that orchestrates everything, and moves + zobrist are services the higher layers (search, training) lean on heavily.

A game in three acts

YINSH unfolds in distinct phases, and the engine encodes them as a GamePhase enum. The phase decides which moves are legal, so almost every method in game_state.py branches on it. Here is the whole arc:

1

RING_PLACEMENT
Players alternate dropping their 5 rings onto empty cells. When all 10 are down, the phase flips automatically.

→
2

MAIN_GAME
Move a ring in a straight line. It leaves a marker behind and flips any markers it jumps over. The real game.

→
3

RING_REMOVAL
Made a run of 5? Remove those markers and lift one of your own rings off the board to score.

→
🏆

Win
Removing 3 rings (POINTS_TO_WIN = 3) ends the game. Steps 2–3 loop until someone gets there.

The counterintuitive bit: removing your own ring is how you win. Every score costs you a piece, so each point makes you slightly weaker — the late game is a tense balance of attack and self-sacrifice. The enum also has ROW_COMPLETION and GAME_OVER states for the in-between bookkeeping moments.

🔁
Markers flip — that's the whole strategy

When a ring slides over a line of markers, every marker it passes flips to the opposite color (board.move_ring: "Move a ring... flipping markers in between"). A single move can turn your opponent's near-complete row into yours, or shatter it entirely. This flip mechanic is why a position can swing wildly from one move — and why the search has to look ahead carefully.

Three directions, not four — and the bug that proved it

On a square grid, a straight line runs in 4 axes. On a hex grid there are only 3. The engine pins this down in constants.py as the single source of truth for "what counts as a straight line." Here's the forward-only set used to scan for runs:

CODE 
# constants.py — the 3 hex axes (forward only)
DIRECTIONS = [
    (0, 1),  # Vertical
    (1, 0),  # Horizontal
    (1, 1),  # Diagonal (matching-sign)
]

# The OTHER diagonal (1, -1) is a PSEUDO-diagonal:
# it is NOT a real line on the YINSH board.
PLAIN ENGLISH

A straight line on this board can run in exactly three ways.

Straight up the column.

Straight across the row.

Diagonally — but only the diagonal where both numbers move the same way (both up, or both down).

The "opposite-sign" diagonal (across one, down the other) looks like a line on a square grid, but in this hex coordinate system it connects cells that aren't actually neighbors.

For the full set of 6 directions (both ways along each axis), the engine has HEX_DIRECTIONS and the membership set HEX_LINE_AXES. Ring movement uses those; run detection uses the 3 forward DIRECTIONS so it walks each axis only once.

🐛
The pseudo-diagonal bug

An earlier version of find_marker_rows hard-coded the (-1, 1) direction into its scan. On a real hex board that's not a line at all, so the engine "found" rows that didn't exist — phantom wins. The fix (April 2026) deleted the inline list and made DIRECTIONS the one place straight-lines are defined. The lesson is universal: when the same geometric truth is re-typed in multiple files, one copy will drift. Centralize it.

Who won? And what about the edge cases?

The referee's verdict lives in game_state.py. It's mostly a score check, but it also handles a subtle case: a player who has no legal moves loses, even without anyone reaching 3 points. That's a stalemate.

CODE 
def get_winner(self):
    if self.white_score >= 3:
        return Player.WHITE
    if self.black_score >= 3:
        return Player.BLACK
    if self.is_stalemate():
        return self.current_player.opponent
    return None
PLAIN ENGLISH

First, the simple case: did White remove 3 rings? They win.

Same check for Black.

Nobody at 3 yet — but is the player to move completely stuck?

If so, they lose: the opponent is the winner.

Otherwise the game is still going — return None (no winner yet).

Note the performance seam: is_terminal() is kept O(1) (just "are we in GAME_OVER?") because MCTS calls it on every node, while the expensive is_stalemate() — which enumerates all legal moves — is only called once per turn at game-ending time. Another careful detail: _handle_marker_removal is atomic. It validates all five target markers first, then removes them, so a malformed move can never leave the board half-updated.

03

Teaching the Computer to See the Board

How a YINSH position becomes numbers, and how the neural network turns those numbers into "where to play" and "who's winning."

A neural net only eats numbers

The neural network at the heart of this project doesn't know what a ring or a marker is. It doesn't see a hexagonal board. It does exactly one thing: take a big grid of numbers in, and push a different set of numbers out. So before the network can have an opinion about a position, we have to hand it the board as numbers.

The translation happens in yinsh_ml/utils/encoding.py, in a class called StateEncoder. Its encode_state method takes a live GameState and produces a tensor of shape (6, 11, 11): six stacked 11×11 grids, one number in every cell.

Why 11×11? The YINSH board is an 11×11 hex grid (85 real playing positions live inside it). And why six grids? Because a single grid can only say one thing per cell — "is there something here?" — but we need to track several different things at once. That's what channels are for: each of the six grids is a separate "layer" recording one specific fact about every cell.

🎛️
Two encoders, one flag

Six channels is the basic encoding. There's also a 15-channel enhanced encoder (EnhancedStateEncoder) that adds pre-computed hints like row threats, ring mobility and score differential. Which one you get is chosen by a single flag — use_enhanced_encoding on the NetworkWrapper — that plumbs all the way down to input_channels=6 or 15 on the network. This module sticks to the 6-channel basic encoding.

What lives in each of the six channels

Here's the whole encoding, channel by channel. Notice the first four are about pieces, the fifth is about which moves are even legal, and the sixth quietly smuggles in two extra facts (the phase, and whose turn it is).

0
Current player's rings

A 1.0 wherever the side-to-move has a ring, 0.0 everywhere else.

1
Opponent's rings

Same idea, but for the other player's rings.

2
Current player's markers

Markers (the flippable pieces) belonging to the side-to-move.

3
Opponent's markers

The other player's markers.

4
Valid-moves mask

A 1.0 on every cell that is a legal source to move/place from right now — a cheat-sheet of where action is possible.

5
Game phase (broadcast) + side-to-move sentinel

The same phase number (0.0 / 0.5 / 1.0) written across the whole grid, with one off-board cell — A1 — overwritten with a 0/1 flag for whose turn it is.

The clever bit is channels 0–3. They are side-normalized: channel 0 is always "the player to move," whether that's White or Black. The network never has to learn "am I White or Black today?" — it always sees the world from the mover's point of view. That's the same trick chess engines use to halve what they need to learn.

💡
The A1 sentinel: a smuggled bit

Because channels 0–3 hide the real colors, the decoder needs a way to remember whose turn it actually is. The encoder stashes that one bit at cell A1 on channel 5. A1 is off the real board (column A only has rows 2–5), so it never collides with a real piece — a free hiding spot for metadata. decode_state reads it back to un-normalize the colors.

The encoder, line by line

Here's the real heart of encode_state. On the left is the actual code; on the right is what each piece is really doing.

CODE 
state = np.zeros((6, 11, 11))
is_white = (current_player == WHITE)

current_rings = white_rings if is_white
                else black_rings
for pos in current_rings:
    state[0, row, col] = 1.0

for move in get_valid_moves():
    state[4, src_row, src_col] = 1.0

phase = phase.value / (len(GamePhase)-1)
state[5] = phase
state[5, 0, 0] = 0.0 if is_white else 1.0
PLAIN ENGLISH

Start with six empty 11×11 grids, all zeros.

Figure out who's to move so we can normalize by side.

"Current player's rings" means White's rings if White is to move, otherwise Black's — the network always sees the mover first.

Stamp a 1.0 into channel 0 at each of those ring positions.

Ask the game for every legal move, and light up channel 4 at every cell you could move from — a built-in legality hint.

Squash the phase (placement / main / removal) into 0.0, 0.5 or 1.0 and paint it across all of channel 5.

Finally, overwrite the off-board A1 cell with a single bit recording whose turn it is — the metadata smuggled in earlier.

(Channels 1, 2 and 3 are filled exactly like channel 0 — opponent rings, current markers, opponent markers — just omitted above to keep it readable.)

From a stack of grids to two answers

That 6×11×11 tensor goes into YinshNetwork (in yinsh_ml/network/model.py). It's a CNN in the AlphaZero style. First a stack of residual blocks and attention blocks chew on the board, each convolution learning richer patterns — runs forming, rings boxed in, threats building.

Then the network splits into two heads — two separate small sub-networks reading from the same shared body. One predicts what to do, the other predicts how it's going.

🎯

Policy head

Outputs 7433 numbers — one "score per move slot" across every move the game could ever express. After softmax these become a probability distribution: "I'd play here 40% of the time, there 12%..." It's the network's gut instinct about good moves. Raw, pre-softmax outputs are called logits.

⚖️

Value head

Outputs a single number between −1 and +1: who's winning from here. +1 means the side-to-move is winning, −1 means losing, 0 is a dead heat. The final squashing uses tanh (or a softmax over outcome classes), so the answer always stays in range.

Why 7433? It's not a round number someone picked — it's computed: 85 ring-placement slots + 7140 ring-move slots + 85 ring-removal slots + 123 marker-removal lines. The single source of truth is StateEncoder().total_moves, and the network reads that value rather than hardcoding it.

⚠️
7433 is a hard contract

A saved model's policy head is exactly 7433 wide, and the slot-to-move mapping is baked in. If you load a checkpoint whose policy size doesn't match the current encoder, NetworkWrapper.load_model hard-fails with an error — it refuses to silently load a mismatched head and play garbage. (Older checkpoints used 8390 slots before a move-encoding fix; those are simply not load-compatible.)

04

How the AI Thinks

The net has a hunch about each move — but a hunch is not a plan. So the AI plays out many what-ifs.

Imagine staring at a YINSH board. Your gut says "that ring looks strong" — but a strong-looking move can still lose three turns later. Good players do not trust the gut alone; they play the position forward in their head: if I go here, they go there, then I can do this. That forward simulation is what we call game-tree search, and it is the whole job of the code in this module.

YinshML uses two different search engines, suited to two different situations:

🎲
MCTS — the explorer

Runs many guided "rollouts" through promising lines, building a search tree shaped by the neural network and/or heuristics. Lives in yinsh_ml/training/self_play.py (class MCTS). This is the engine used to generate training data.

✅
Negamax + alpha-beta — the calculator

A classical brute-force search that examines moves to a fixed depth, pruning lines it can prove are irrelevant. Lives in yinsh_ml/agents/heuristic_agent.py. This is the pure, network-free opponent.

The calculator uses negamax with alpha-beta pruning — it scores leaves with the heuristic evaluator and cuts off branches that cannot beat what it has already found.

💡
One engine, not two MCTS

There used to be a second, broken MCTS in search/mcts.py that never actually reached the network. It was deleted. Today there is exactly one MCTS engine: MCTS.search_batch() is the batched path used in training, and MCTS.search() is the single-position path.

The clever part is the SELECT step. The rule used to choose a child is called PUCT. It combines a "what we have learned" term (Q, the average value seen) with a "what we have not yet tried" bonus (U, biggest for high-prior, low-visit children). Early on, U dominates and the search spreads out; as visits accumulate, Q takes over and the search focuses.

Reading the selection formula

Here is the heart of _select_action in the real MCTS code — the vectorized PUCT score it computes for every child at once. Do not worry about NumPy; read the right-hand column.

CODE 
scaled_q = self.value_weight * q_values
sqrt_parent = np.sqrt(parent_visit_count)
u_visited = c_puct * priors * sqrt_parent / (1.0 + visits)
u_unvisited = c_puct * priors * sqrt_parent_eps
u_values = np.where(visited_mask, u_visited, u_unvisited)
scores = scaled_q + u_values + noise
PLAIN ENGLISH

Q term: how good has this child looked across all the times we visited it.

The exploration bonus grows with the square root of how often we visited the parent.

U term for visited children: high when the network gave it a strong prior, but it shrinks each time we visit it (the 1 + visits divisor).

A child we have never tried gets a clean exploration bonus with no shrink.

Pick the right formula per child: visited ones shrink, fresh ones do not.

Final score = exploit (Q) + explore (U) + a tiny random tiebreaker. The child with the highest score wins this step.

One subtlety worth a glossary stop: First-Play Urgency. A child with zero visits has no average value to use, so MCTS estimates one: it takes the parent value and subtracts a small penalty (fpu_reduction, default 0.25). This stops the search from wandering into every untried branch at once.

The 7 heuristic features

When the search needs to score a position without the network (in pure_heuristic mode, or as the calculator agent leaf evaluation), it uses a hand-tuned heuristic evaluator in yinsh_ml/heuristics/evaluator.py. It scores a board as a weighted sum of seven features, learned from analysis of 100K+ games:

🏆

Completed Runs Differential 0.239

My finished 5-in-a-rows minus yours. The single most important signal — it is closest to actually winning.

🎯

Ring Centrality 0.211

How central my rings sit. Central rings reach more of the board and create more threats.

↔

Ring Spread 0.187

How well my rings are distributed rather than clumped, keeping options open across the board.

🔬

Potential Runs 0.171

How many near-complete rows I am building — threats that could become completed runs soon.

🔗

Connected Marker Chains 0.086

Runs of my markers already linked together, the raw material for future rows.

🔁

Ring Mobility 0.071

How many moves my rings have available. More mobility means more flexibility.

📍

Edge Proximity 0.036

A small penalty for hugging the board edge, where rings are cramped. The least influential feature.

These weights are phase-specific. The weight_manager.py swaps in different weight sets for the opening, middlegame, and endgame — mobility matters early, completed runs and center control matter late.

⚠️
The offense-only equilibrium trap

Look closely: all seven features are attack-oriented and all differential (mine minus yours). There is no explicit "block their threat" feature. Defense is only captured implicitly — under negamax, blocking shows up because the opponent gains lower the differential a few plies ahead. This works at search depth ≥3, but at depth 1–2 the search cannot see far enough and defense can collapse, with both sides racing to attack while ignoring each other. The live game viewer exists partly to catch exactly this failure signature.

Not searching the same position twice

Different move orders often reach the same board. Re-searching it is wasted work. A transposition table (in yinsh_ml/search/transposition_table.py) caches each searched position so the calculator agent can reuse the result instantly.

The cache key comes from Zobrist hashing. Crucially, ZobristHasher.hash_state mixes in three things, not just the pieces:

CODE 
current = self.hash_board(game_state.board)
current = self.toggle_side_to_move(game_state.current_player, current)
current = self.toggle_phase(game_state.phase, current)
return current
PLAIN ENGLISH

Start from the hash of the piece layout (XOR a random number for every piece on every square).

Mix in whose turn it is — the same layout is a different position if it is the other player to move.

Mix in the game phase — placement, main game, and ring-removal have different legal moves even with identical pieces.

The result is one 64-bit key. Two truly-different positions almost never collide.

📊
Depth-preferred, and it pays off

The table holds 220 (~1M) entries by default. When a slot is taken, it keeps the entry from the deeper search (depth-preferred replacement), since deeper results are more valuable. Typical hit rates land at 60–80% during tree search — most positions are answered from cache.

05

Learning From Experience

The AI just played thousands of games against itself. Now what?

Playing against a mirror

By now the agent can search a position and pick a move. But a player that never learns from its games never improves. The trick at the heart of this whole project is self-play: the network plays full games against a copy of itself, and every position it visits becomes a tiny lesson it can study afterward.

This is the loop that powers AlphaZero-style systems. There are no human game records, no labeled "correct moves" handed down by an expert. The data the network learns from is data it generated itself — and because each new version plays a little better than the last, the lessons keep getting harder and better. The system pulls itself up by its own bootstraps.

Three things happen in order: generate games, turn each position into a training example, then nudge the network toward those examples. Let's walk through the lifecycle of a single position as it travels from the board to a gradient update.

💡
Why self-play works at all

The search (MCTS) is smarter than the raw network because it actually looks ahead. So the network learns to imitate a stronger version of itself — the search results — rather than just copying what it already believed. That gap is where improvement comes from.

The lifecycle of one position

Each game produces a stream of board positions. For every position the system records three things together — this trio is called an experience tuple: the state (the board, encoded as tensors), the policy target (which moves were worth considering), and the value target (did this side end up winning).

The actors in that loop map directly to files: yinsh_ml/self_play/game_runner.py drives the games (SelfPlayRunner), yinsh_ml/training/self_play.py turns positions into tuples, yinsh_ml/self_play/data_storage.py writes Parquet, and yinsh_ml/training/trainer.py holds the replay buffer and the training step.

📁
Parquet vs JSON: about 500× smaller

Games are stored in Parquet, not JSON. A single game lands at roughly 3 KB versus about 1.5 MB as JSON — a ~500× reduction. That is the difference between storing a hundred games and storing a hundred thousand.

Where do the two targets come from?

The state is the easy part: it is just the board encoded as a stack of tensors (covered in Module 3). The two targets are where the cleverness lives, and they come from very different places.

The policy target is the MCTS visit-count distribution: a probability spread over moves, where moves the search explored more get more weight. Crucially this is a better policy than the raw network's first guess, because the search actually looked ahead before deciding what to explore. That is what the network chases.

The value target is the eventual game result from that position's point of view — a number reflecting whether the side to move went on to win, lose, or draw. A position from a game you won is labeled positive; one from a game you lost is labeled negative. Over millions of positions, the network learns to look at a board and estimate "am I winning here?"

🗺️
State — the question

The encoded board: rings, markers, valid moves, phase. What the network sees as input.

🎯
Policy target — the move answer

MCTS visit counts as a probability distribution. "These were the moves worth thinking about."

🏆
Value target — the outcome answer

The eventual game result from this side's perspective. "This position led to a win/loss."

✂️
Why subsample long games?

In GameExperience.add_game_experience, games longer than 100 moves are subsampled: keep the first 20 and last 20 positions, randomly sample ~30 from the middle. A 200-move game has a lot of near-identical mid-game positions — keeping all of them would flood the buffer with redundant, correlated examples and crowd out variety. Trimming keeps the data diverse.

One network, two heads, two losses

The network has two output heads, so training uses two losses that get summed together. The policy head learns with cross-entropy against the MCTS distribution. The value head learns to match the game outcome. Here is the real policy loss from trainer.py.

CODE 
scaled_logits = pred_logits / self.temperature
log_probs = F.log_softmax(scaled_logits, dim=1)
policy_loss = -(target_probs * log_probs).sum(dim=1).mean()

# value head, classification mode
log_probs = F.log_softmax(value_logits_fp32, dim=1)
ce_loss = -(target_dist * log_probs).sum(dim=1).mean()
value_loss = ce_loss + discrimination_loss
PLAIN ENGLISH

Soften the raw move scores with a temperature knob first.

Turn them into log-probabilities the network is predicting.

Cross-entropy: punish the network wherever its move probabilities disagree with the MCTS visit counts. This is the policy loss.

 

Now the value head, which predicts the outcome.

Same trick: log-probabilities, this time over discrete outcome buckets.

Compare against the game result and add a nudge that rewards confident (non-zero) predictions. This is the value loss.

Both losses share one forward pass and are added into a single total_loss that backpropagates once. The shared body of the network therefore learns from both signals at the same time — "which move" and "am I winning" — which is exactly why one network can do both jobs. (The value head here uses a classification framing over outcome buckets; a simpler setup would use plain MSE toward the outcome.)

Who actually picks the moves?

Self-play needs a move-selection policy — the thing that decides what to play at each turn. The codebase ships four of them in yinsh_ml/self_play/policies.py, ranging from "pick anything" to "search hard, guided by the network." You mix them depending on what kind of training data you want.

🎲

Random

RandomMovePolicy — picks a legal move at random (with a small rule-based nudge). The baseline: cheap, weak, useful for sanity checks and bootstrapping.

📐

Heuristic

HeuristicPolicy — negamax search over the 7 learned features. No neural net needed; strong and fully on the CPU.

🧠

MCTS

MCTSPolicy — tree search guided by the network's policy prior and value estimate. This is the policy that produces the rich visit-count targets training depends on.

🔀

Adaptive

AdaptivePolicy — switches between the others based on training progress. Cheap policies early when the net is weak, stronger search later.

With the data lifecycle and the loss in hand, test the most important idea: where the training targets actually come from.

In an experience tuple, where do the policy target and value target come from?

So the engine generates games, distills each position into a (state, policy, value) lesson, stores it compactly, and replays it into a two-headed gradient step — over and over, getting stronger each cycle. But something has to orchestrate all of this: decide when to stop self-play and start training, measure whether the new network is actually better, and run the tournaments that produce ELO scores. Orchestration, Evaluation & Finding Your Way

You now know the moving parts — here is the conductor that drives them, and how to jump in and make your first change.

Meet the conductor

Over the last five modules you met the players in the orchestra: the game engine, the encoder and network, the search, and the self-play training loop. This module introduces the conductor that keeps them all in time — and then hands you the baton.

That conductor is the TrainingSupervisor in yinsh_ml/training/supervisor.py. It runs the whole show in a loop: generate a batch of self-play games, train the network on those games, then evaluate the freshly trained network to decide whether it is actually better than the best model so far. Only if it passes does it get promoted to become the new best.

It is a big file — its own header admits it is ~2,500 lines and overdue for a split — because it has absorbed orchestration, the training loop, tournament evaluation, checkpoint management, and metrics logging. Do not let the size scare you; the lifecycle it runs is simple to picture.

🎲
1. Self-play

Run MCTS-guided games against itself to produce fresh (state, policy, value) training examples.

🧠
2. Train

Update the network on the new data — policy head via cross-entropy, value head via MSE.

🏆
3. Evaluate & promote

Play the new network against the current best in a tournament, track results with ELO, and keep the winner.

Step 3 of the conductor's loop — deciding whether a freshly trained network is actually better — is the subtle part. You cannot just trust a lower training loss; the only honest test is making the two networks play. Here is what that looks like in code terms:

CODE 

candidate = train(network, new_games)
stats = tournament.play(candidate, best_model)
elo = elo_tracker.update(stats)
if passes_gate(stats.win_rate):
    best_model = candidate
PLAIN ENGLISH

Train a candidate network on the games we just generated.

Make the candidate play many games against the current champion (yinsh_ml/utils/tournament.py), alternating colors.

Feed the win/loss/draw record into the rating system (yinsh_ml/utils/elo_manager.py) to get an updated strength number.

Apply the promotion gate: did the candidate win convincingly enough — not just by luck — to be trusted?

If yes, the candidate becomes the new best model. If not, we keep the champion and try again next iteration.

The real supervisor uses a statistical Wilson gate on the win rate (so a lucky streak in a few games is not enough) and tracks ratings over time. A temporary ELO dip between iterations is normal and expected — the model needs to explore worse-looking moves to find better strategies, and recovers with continued training.

💡
Debugging tip: check the cache, then the pools

If search feels slow, inspect the transposition-table hit rate via agent.last_search_stats['transposition_table_metrics']['hit_rate']. A healthy rate is roughly 60–80%; below 50%, grow the table or clear it between games. For memory pressure, enable the GameStatePool and TensorPool from yinsh_ml/memory/ to cut allocation churn.
