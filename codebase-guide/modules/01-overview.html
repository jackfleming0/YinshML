What YinshML Actually Does

A program that teaches itself to play a board game by playing against itself, millions of times, and keeping notes on what worked.

Start with one command

Before any architecture diagrams, here is the thing you will actually run:

python scripts/run_training.py Kicks off the whole learning loop: play games, learn from them, repeat.
python run_large_scale_selfplay.py Just generates game data, without the learning step. Useful for building a dataset.

That first command is the front door to this entire codebase. YinshML is an reinforcement learning framework for YINSH, an abstract strategy board game played on a hexagonal grid. The goal is to produce an agent that plays YINSH extremely well.

The twist: it learns with no human games at all. It is built in the style of AlphaZero — the agent starts knowing only the rules, plays against itself over and over, and gradually gets better by noticing which moves tend to lead to wins.

💡
The whole idea in one sentence

Imagine a player who becomes superhuman purely by playing themselves a few million times and writing down, after every game, which positions turned out well. No coach, no opening book, no human games. That feedback loop is the entire system — everything in this repo exists to make that loop fast and stable.

Two pieces of jargon do all the heavy lifting here. A policy is the network saying "here is how good each possible move looks." A value is the network saying "from this position, I think this side is winning by about this much." The whole point of training is to make both predictions sharper over time.

The six subsystems

The loop above is orchestrated across six major parts of the codebase. Each card is a chunk of the repo you will meet in later modules:

🎲

Game Engine

The rules of YINSH in code: the board, legal moves, when the game is over, who won. Everything else sits on top of this. yinsh_ml/game/

🧠

Neural Network

Looks at a board and outputs two things: a policy (which moves look good) and a value (who is winning). This is the part that actually learns. yinsh_ml/network/

🔍

Search / MCTS

Looks ahead by simulating many possible futures, guided by the network, to pick a stronger move than the network alone would. yinsh_ml/training/self_play.py

📐

Heuristics

Hand-crafted scoring rules (center control, marker chains, etc.) learned from 100K games. They give the search good instincts before the network is strong. yinsh_ml/heuristics/

♟

Self-Play

Runs games of the agent against itself and records every position as training data. The factory that produces experience. yinsh_ml/self_play/

🎛️

Training Supervisor

The conductor. Runs self-play, then training, then evaluation, then repeats — wiring all the pieces above into one loop. yinsh_ml/training/supervisor.py

The Training Supervisor is the one to anchor on. Its single method train_iteration() is one full turn of the loop, and its own docstring spells it out: "One full self-play → training → evaluation → model-selection cycle."

From command to loop, in code

Here is the actual entry point — the tail of scripts/run_training.py. Notice how little there is: it builds the supervisor, then calls one method in a loop.

CODE 
supervisor = TrainingSupervisor(
    network=network,
    mcts_simulations=int(sp.get("num_simulations", 96)),
    mode_settings=mode_settings,
)

for it in range(start_iteration, num_iterations):
    summary = supervisor.train_iteration(
        num_games=games_per_iteration,
        epochs=epochs_per_iteration,
    )
PLAIN ENGLISH

Build the conductor that runs everything, handing it the freshly built neural network.

Tell its look-ahead search how hard to think per move — here, 96 simulated futures.

Pass in all the knobs from the config file (learning rate, how many games, and so on).

Now just repeat the loop, once per iteration, for as many rounds as configured (default 50).

Each call to train_iteration is one full turn of the wheel: play games, learn, get tested. That single line is the heartbeat of the whole project.

Those knobs live in plain YAML at configs/training.yaml. For example, num_iterations: 50 sets how many times the loop runs, games_per_iteration: 50 sets how many self-play games feed each round, and promotion_threshold: 0.55 sets the bar a new network must clear in the tournament to be crowned the new best. You tune the experiment by editing that file, not the code.

📊
How "is the new one actually better?" gets answered

After training, the new network plays a head-to-head tournament against the reigning champion. Results feed an ELO rating so you can track strength across versions on one scale. The network is only kept if it wins clearly enough — that gate is what makes the loop a ratchet that mostly goes up.

That is the 10,000-foot view: one command, one loop, six subsystems, getting stronger each pass. Everything rests on the rules of the game itself — so Module 2 dives into the Game Engine: the hexagonal board, how moves and captures work, and how a YINSH position is represented in code.
