The Board and the Rules

How a hexagonal board game becomes clean, testable Python — pieces, phases, and the one bug that hid in plain sight.

Picture the board first

Before a single line of code makes sense, you need the mental picture. YINSH is played on a hexagonal grid — not a square chessboard. The code models it as an 11×11 coordinate space, but the playing surface is a diamond cut out of that square: only 85 of the 121 cells are real. The corners are empty.

Each column (lettered A through K) holds a different number of valid rows, and those counts form the diamond shape: [4, 7, 8, 9, 10, 9, 10, 9, 8, 7, 4]. Column A only has rows 2–5; the fat middle columns reach all the way across. This is why you'll constantly see is_valid_position() guarding the code — most of the coordinate plane simply isn't on the board.

Two players, WHITE and BLACK, share the board. Each owns two kinds of pieces: rings (the things you actually move) and markers (tokens that get left behind and flip color). Hold those two ideas — rings move, markers accumulate — and the rest of the engine falls into place.

💡
Why integers for players?

Player.WHITE = 1 and Player.BLACK = -1 isn't arbitrary. Encoding the two sides as +1 / -1 means "the other player" is just multiplication by −1, and a board can be flipped to the current player's perspective by sign math instead of branching. This tiny choice ripples all the way up into how the neural network sees positions in Module 3.

Six files, one game engine

Everything about the rules lives in yinsh_ml/game/. There's no AI here yet — this is the pure, deterministic referee that the search algorithms and neural network sit on top of. Each file has a single, sharp job:

yinsh_ml/game/ The rules of YINSH, as plain Python — no machine learning
constants.py Board geometry & rule numbers: the 85 valid positions, the hex directions, 5 rings per player, 5 markers per row, 3 rows to win
types.py The vocabulary: Move, MoveType, GamePhase enums — small, frozen dataclasses everything else passes around
board.py Where the pieces actually are. Places/moves pieces, flips markers when a ring jumps them, finds completed rows
game_state.py The referee: whose turn it is, the current phase, move history, win/stalemate detection, applying a move
moves.py Move generation — given a state, enumerate every legal move a player could make right now
zobrist.py A fast fingerprint of a position (a single integer hash) so the search can recognize "seen this before" instantly

A useful way to read the dependency arrow: constants and types define the nouns and numbers, board holds the physical state, game_state is the rules engine that orchestrates everything, and moves + zobrist are services the higher layers (search, training) lean on heavily.

A game in three acts

YINSH unfolds in distinct phases, and the engine encodes them as a GamePhase enum. The phase decides which moves are legal, so almost every method in game_state.py branches on it. Here is the whole arc:

1

RING_PLACEMENT
Players alternate dropping their 5 rings onto empty cells. When all 10 are down, the phase flips automatically.

→
2

MAIN_GAME
Move a ring in a straight line. It leaves a marker behind and flips any markers it jumps over. The real game.

→
3

RING_REMOVAL
Made a run of 5? Remove those markers and lift one of your own rings off the board to score.

→
🏆

Win
Removing 3 rings (POINTS_TO_WIN = 3) ends the game. Steps 2–3 loop until someone gets there.

The counterintuitive bit: removing your own ring is how you win. Every score costs you a piece, so each point makes you slightly weaker — the late game is a tense balance of attack and self-sacrifice. The enum also has ROW_COMPLETION and GAME_OVER states for the in-between bookkeeping moments.

🔁
Markers flip — that's the whole strategy

When a ring slides over a line of markers, every marker it passes flips to the opposite color (board.move_ring: "Move a ring... flipping markers in between"). A single move can turn your opponent's near-complete row into yours, or shatter it entirely. This flip mechanic is why a position can swing wildly from one move — and why the search has to look ahead carefully.

Three directions, not four — and the bug that proved it

On a square grid, a straight line runs in 4 axes. On a hex grid there are only 3. The engine pins this down in constants.py as the single source of truth for "what counts as a straight line." Here's the forward-only set used to scan for runs:

CODE 
# constants.py — the 3 hex axes (forward only)
DIRECTIONS = [
    (0, 1),  # Vertical
    (1, 0),  # Horizontal
    (1, 1),  # Diagonal (matching-sign)
]

# The OTHER diagonal (1, -1) is a PSEUDO-diagonal:
# it is NOT a real line on the YINSH board.
PLAIN ENGLISH

A straight line on this board can run in exactly three ways.

Straight up the column.

Straight across the row.

Diagonally — but only the diagonal where both numbers move the same way (both up, or both down).

The "opposite-sign" diagonal (across one, down the other) looks like a line on a square grid, but in this hex coordinate system it connects cells that aren't actually neighbors.

For the full set of 6 directions (both ways along each axis), the engine has HEX_DIRECTIONS and the membership set HEX_LINE_AXES. Ring movement uses those; run detection uses the 3 forward DIRECTIONS so it walks each axis only once.

🐛
The pseudo-diagonal bug

An earlier version of find_marker_rows hard-coded the (-1, 1) direction into its scan. On a real hex board that's not a line at all, so the engine "found" rows that didn't exist — phantom wins. The fix (April 2026) deleted the inline list and made DIRECTIONS the one place straight-lines are defined. The lesson is universal: when the same geometric truth is re-typed in multiple files, one copy will drift. Centralize it.

Who won? And what about the edge cases?

The referee's verdict lives in game_state.py. It's mostly a score check, but it also handles a subtle case: a player who has no legal moves loses, even without anyone reaching 3 points. That's a stalemate.

CODE 
def get_winner(self):
    if self.white_score >= 3:
        return Player.WHITE
    if self.black_score >= 3:
        return Player.BLACK
    if self.is_stalemate():
        return self.current_player.opponent
    return None
PLAIN ENGLISH

First, the simple case: did White remove 3 rings? They win.

Same check for Black.

Nobody at 3 yet — but is the player to move completely stuck?

If so, they lose: the opponent is the winner.

Otherwise the game is still going — return None (no winner yet).

Note the performance seam: is_terminal() is kept O(1) (just "are we in GAME_OVER?") because MCTS calls it on every node, while the expensive is_stalemate() — which enumerates all legal moves — is only called once per turn at game-ending time. Another careful detail: _handle_marker_removal is atomic. It validates all five target markers first, then removes them, so a malformed move can never leave the board half-updated.
