03

Teaching the Computer to See the Board

How a YINSH position becomes numbers, and how the neural network turns those numbers into "where to play" and "who's winning."

A neural net only eats numbers

The neural network at the heart of this project doesn't know what a ring or a marker is. It doesn't see a hexagonal board. It does exactly one thing: take a big grid of numbers in, and push a different set of numbers out. So before the network can have an opinion about a position, we have to hand it the board as numbers.

The translation happens in yinsh_ml/utils/encoding.py, in a class called StateEncoder. Its encode_state method takes a live GameState and produces a tensor of shape (6, 11, 11): six stacked 11×11 grids, one number in every cell.

Why 11×11? The YINSH board is an 11×11 hex grid (85 real playing positions live inside it). And why six grids? Because a single grid can only say one thing per cell — "is there something here?" — but we need to track several different things at once. That's what channels are for: each of the six grids is a separate "layer" recording one specific fact about every cell.

🎛️
Two encoders, one flag

Six channels is the basic encoding. There's also a 15-channel enhanced encoder (EnhancedStateEncoder) that adds pre-computed hints like row threats, ring mobility and score differential. Which one you get is chosen by a single flag — use_enhanced_encoding on the NetworkWrapper — that plumbs all the way down to input_channels=6 or 15 on the network. This module sticks to the 6-channel basic encoding.

What lives in each of the six channels

Here's the whole encoding, channel by channel. Notice the first four are about pieces, the fifth is about which moves are even legal, and the sixth quietly smuggles in two extra facts (the phase, and whose turn it is).

0
Current player's rings

A 1.0 wherever the side-to-move has a ring, 0.0 everywhere else.

1
Opponent's rings

Same idea, but for the other player's rings.

2
Current player's markers

Markers (the flippable pieces) belonging to the side-to-move.

3
Opponent's markers

The other player's markers.

4
Valid-moves mask

A 1.0 on every cell that is a legal source to move/place from right now — a cheat-sheet of where action is possible.

5
Game phase (broadcast) + side-to-move sentinel

The same phase number (0.0 / 0.5 / 1.0) written across the whole grid, with one off-board cell — A1 — overwritten with a 0/1 flag for whose turn it is.

The clever bit is channels 0–3. They are side-normalized: channel 0 is always "the player to move," whether that's White or Black. The network never has to learn "am I White or Black today?" — it always sees the world from the mover's point of view. That's the same trick chess engines use to halve what they need to learn.

💡
The A1 sentinel: a smuggled bit

Because channels 0–3 hide the real colors, the decoder needs a way to remember whose turn it actually is. The encoder stashes that one bit at cell A1 on channel 5. A1 is off the real board (column A only has rows 2–5), so it never collides with a real piece — a free hiding spot for metadata. decode_state reads it back to un-normalize the colors.

The encoder, line by line

Here's the real heart of encode_state. On the left is the actual code; on the right is what each piece is really doing.

CODE 
state = np.zeros((6, 11, 11))
is_white = (current_player == WHITE)

current_rings = white_rings if is_white
                else black_rings
for pos in current_rings:
    state[0, row, col] = 1.0

for move in get_valid_moves():
    state[4, src_row, src_col] = 1.0

phase = phase.value / (len(GamePhase)-1)
state[5] = phase
state[5, 0, 0] = 0.0 if is_white else 1.0
PLAIN ENGLISH

Start with six empty 11×11 grids, all zeros.

Figure out who's to move so we can normalize by side.

"Current player's rings" means White's rings if White is to move, otherwise Black's — the network always sees the mover first.

Stamp a 1.0 into channel 0 at each of those ring positions.

Ask the game for every legal move, and light up channel 4 at every cell you could move from — a built-in legality hint.

Squash the phase (placement / main / removal) into 0.0, 0.5 or 1.0 and paint it across all of channel 5.

Finally, overwrite the off-board A1 cell with a single bit recording whose turn it is — the metadata smuggled in earlier.

(Channels 1, 2 and 3 are filled exactly like channel 0 — opponent rings, current markers, opponent markers — just omitted above to keep it readable.)

From a stack of grids to two answers

That 6×11×11 tensor goes into YinshNetwork (in yinsh_ml/network/model.py). It's a CNN in the AlphaZero style. First a stack of residual blocks and attention blocks chew on the board, each convolution learning richer patterns — runs forming, rings boxed in, threats building.

Then the network splits into two heads — two separate small sub-networks reading from the same shared body. One predicts what to do, the other predicts how it's going.

🎯

Policy head

Outputs 7433 numbers — one "score per move slot" across every move the game could ever express. After softmax these become a probability distribution: "I'd play here 40% of the time, there 12%..." It's the network's gut instinct about good moves. Raw, pre-softmax outputs are called logits.

⚖️

Value head

Outputs a single number between −1 and +1: who's winning from here. +1 means the side-to-move is winning, −1 means losing, 0 is a dead heat. The final squashing uses tanh (or a softmax over outcome classes), so the answer always stays in range.

Why 7433? It's not a round number someone picked — it's computed: 85 ring-placement slots + 7140 ring-move slots + 85 ring-removal slots + 123 marker-removal lines. The single source of truth is StateEncoder().total_moves, and the network reads that value rather than hardcoding it.

⚠️
7433 is a hard contract

A saved model's policy head is exactly 7433 wide, and the slot-to-move mapping is baked in. If you load a checkpoint whose policy size doesn't match the current encoder, NetworkWrapper.load_model hard-fails with an error — it refuses to silently load a mismatched head and play garbage. (Older checkpoints used 8390 slots before a move-encoding fix; those are simply not load-compatible.)

So now the network can look at any position and instantly offer two opinions: a ranked list of moves, and a guess at who's ahead. But a single gut reaction isn't enough to play well — strong play comes from thinking ahead, trying moves, and seeing where they lead.
