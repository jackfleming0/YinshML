Meet the conductor

Over the last five modules you met the players in the orchestra: the game engine, the encoder and network, the search, and the self-play training loop. This module introduces the conductor that keeps them all in time — and then hands you the baton.

That conductor is the TrainingSupervisor in yinsh_ml/training/supervisor.py. It runs the whole show in a loop: generate a batch of self-play games, train the network on those games, then evaluate the freshly trained network to decide whether it is actually better than the best model so far. Only if it passes does it get promoted to become the new best.

It is a big file — its own header admits it is ~2,500 lines and overdue for a split — because it has absorbed orchestration, the training loop, tournament evaluation, checkpoint management, and metrics logging. Do not let the size scare you; the lifecycle it runs is simple to picture.

🎲
1. Self-play

Run MCTS-guided games against itself to produce fresh (state, policy, value) training examples.

🧠
2. Train

Update the network on the new data — policy head via cross-entropy, value head via MSE.

🏆
3. Evaluate & promote

Play the new network against the current best in a tournament, track results with ELO, and keep the winner.

Your first hour with the repo

Reading is good; running is better. Here is a concrete on-ramp from a fresh clone to watching a live training run. Every command below comes straight from the repo docs.

1
Set up the environment

Create a virtualenv, install dependencies, and install the package in editable mode so your code changes take effect immediately.

python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
pip install -e .
2
Run the tests

Confirm a green baseline before changing anything. Tests live in tests/ and yinsh_ml/tests/; the slow ones are marked @pytest.mark.slow, so skip them for a fast first pass.

pytest -m "not slow"
3
Generate some self-play data

Produce a batch of games the trainer can learn from. The output is compact Parquet (~3KB per game).

python run_large_scale_selfplay.py
4
Launch a training run

Hand the whole loop to the supervisor. Point it at a config to control iteration count, encoding type, and more.

python scripts/run_training.py --config configs/my_config.yaml
5
Watch the dashboard

In a second terminal, open the dashboard to watch ELO, win rates, and loss curves move as training proceeds.

python run_dashboard.py

The entry points, and how a model earns promotion

A handful of commands are the doors into the system. Keep this list handy — each one is a different way to drive or inspect the same machinery.

pytest -m "not slow" Run the fast test suite (game logic, search, heuristics, training)
python scripts/run_training.py Start the TrainingSupervisor: self-play, train, evaluate, promote
python run_large_scale_selfplay.py Generate training data via large-scale self-play
yinsh-track --help CLI for experiment tracking (commands in yinsh_ml/cli/commands/)
streamlit run scripts/dashboard_games.py Live game viewer for auditing self-play game by game

Step 3 of the conductor's loop — deciding whether a freshly trained network is actually better — is the subtle part. You cannot just trust a lower training loss; the only honest test is making the two networks play. Here is what that looks like in code terms:

CODE 

candidate = train(network, new_games)
stats = tournament.play(candidate, best_model)
elo = elo_tracker.update(stats)
if passes_gate(stats.win_rate):
    best_model = candidate
PLAIN ENGLISH

Train a candidate network on the games we just generated.

Make the candidate play many games against the current champion (yinsh_ml/utils/tournament.py), alternating colors.

Feed the win/loss/draw record into the rating system (yinsh_ml/utils/elo_manager.py) to get an updated strength number.

Apply the promotion gate: did the candidate win convincingly enough — not just by luck — to be trusted?

If yes, the candidate becomes the new best model. If not, we keep the champion and try again next iteration.

The real supervisor uses a statistical Wilson gate on the win rate (so a lucky streak in a few games is not enough) and tracks ratings over time. A temporary ELO dip between iterations is normal and expected — the model needs to explore worse-looking moves to find better strategies, and recovers with continued training.

💡
Debugging tip: check the cache, then the pools

If search feels slow, inspect the transposition-table hit rate via agent.last_search_stats['transposition_table_metrics']['hit_rate']. A healthy rate is roughly 60–80%; below 50%, grow the table or clear it between games. For memory pressure, enable the GameStatePool and TensorPool from yinsh_ml/memory/ to cut allocation churn.

You made it — now go build

Congratulations: you have walked the whole tour. You know the game engine and its rules, how a board becomes tensors and flows through the network, how MCTS turns a hunch into a plan, how self-play turns play into learning, and now how the TrainingSupervisor conducts it all and decides which model earns promotion.
