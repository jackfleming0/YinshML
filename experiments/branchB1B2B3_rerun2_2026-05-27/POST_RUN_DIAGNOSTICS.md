# B1+B2+B3 RE-RUN #2 — Post-run diagnostics

**Date:** 2026-05-27 (evening, post-SPRT)
**Purpose:** answer two of the four mechanism hypotheses for the iter-3-4
WR drift (52.0 → 50.5 → 45.0), using artifacts already on disk — no
new GPU work.

The pre-diagnostic working priors (logged in `EXPERIMENT_BACKLOG.md`
investigation queue, commit `2642794`):

| # | Mechanism | Prior |
|---|---|---|
| 1 | Heuristic-weight annealing exposing iter_1's noisy value head | ~30% |
| 2 | Adam optimizer state surviving revert | ~15% |
| 3 | Buffer composition / mode collapse | ~5% |
| 4 | Pure noise | ~50% |

## 1. Replay buffer composition — 3-way comparison

Ran `scripts/diagnose_replay_buffers.py` across three buffers:
- D.2 (2026-05-25): 15-ch, **buggy** phase weights, 5 iters
- B1B2B3 original (2026-05-26): 15-ch, **buggy** phase weights, 5 iters
- B1B2B3 RE-RUN #2 (2026-05-27): 15-ch, **fixed** phase weights, 5 iters

### Headline metrics

| Metric | D.2 buggy | B1B2B3 orig buggy | B1B2B3 fixed |
|---|---|---|---|
| Sample count | 100,000 | 100,000 | 100,000 |
| **Stored MAIN_GAME %** | 0.0% | 0.0% | **76.3%** |
| **True MAIN_GAME %** (fixed decoder) | 76.2% | 76.4% | 75.9% |
| Frac dummy/zero rows | 73.8% | 55.8% | 61.7% |
| Policy nonzero mean (real rows) | 8.2 | 6.9 | 7.7 |
| Value mean | +0.0497 | +0.0474 | +0.0522 |

| Value % at zero | 0.0% | 0.0% | 0.0% |
| Inferred game length (mean) | 64.9 | 64.5 | 64.0 |

### Findings

**1. Phase-weight bug confirmed at the data level.** The buggy runs
stored 100% RING_PLACEMENT labels, but applying the fixed
`decode_phase_from_state` to the actual stored state tensors recovers
~76% MAIN_GAME. The bug warped the **sampling weights applied to
training**, not the **gameplay generated by self-play.** This is exactly
the prediction; it's now confirmed directly from the data.

**2. Game-length and overall structure are stable across all three
runs.** Mean inferred game length ~64 moves; move_number distributions
look identical; value means all ~+0.05 (slight white-favored, basically
symmetric). The loop's gameplay didn't drift between buggy and fixed.

**3. One real shift in the fixed run: value distribution is more
decisive.** The fixed-phase B1B2B3 buffer has **14.8% extreme-value
samples (one side wins 3-0)** vs ~11% in the buggy runs — a 37%
relative increase. Possible reads:
- *Positive:* training emphasizes MAIN_GAME properly → network plays
  more decisively in tactical phases → more skewed outcomes.
- *Cautious:* could be early mode-collapse signal (committing to one
  strategy).
Not dramatic enough to be alarming, but worth noting for the next
experiment.

**4. Dummy/zero-policy row fraction is 55-74% across all runs.**
Terminal positions and MCTS-bypass moves. For D1-partial pretraining,
the converter already filters these → effective corpus is ~38K usable
samples per buffer, not 100K.

### Conclusion: Mechanism 3 (buffer composition) — ❌ ELIMINATED

The iter-3-4 drift is NOT explained by buffer composition shifts. All
three runs' buffers look qualitatively similar (modulo phase labels and
the value-decisiveness shift). If buffer drift were the drift
mechanism, we'd expect clear shifts in policy sparsity / value
distribution / game length — we don't.

## 2. Optimizer state on revert — code inspection

**Question:** when the gate reverts (weights reload from `best_model.pt`),
does Adam's `(m, v)` momentum state also reset? Or does it survive and
prime the next iter's training to drift in the same direction the
rejected iter was heading?

### Code path

`supervisor.py:1825-1862` — revert path:

```python
if self.revert_on_gate_failure and self.best_model_path and self.best_model_path.exists():
    ...
    if proceed:
        try:
            self.network.load_model(str(self.best_model_path))
            decision_kind = 'reverted'
            self.logger.info(f"   ✓ Loaded best weights from {self.best_model_path.name}")
            if self.reset_optimizer_on_revert:
                # Momentum was building in the direction of the rejected
                # weights — that direction was bad by definition (gate
                # failed), so resetting prevents the next training step
                # from immediately marching back into the rejected zone.
                self._reinitialize_optimizers()
```

`supervisor.py:2284-2333` — `_reinitialize_optimizers`:

```python
def _reinitialize_optimizers(self):
    ...
    self.trainer.policy_optimizer = optim.Adam(
        policy_params,
        lr=self.trainer._base_policy_lr,
        weight_decay=1e-4
    )
    self.trainer.value_optimizer = optim.SGD(
        value_params,
        lr=self.trainer._base_value_lr,
        momentum=0.9,
        weight_decay=1e-3
    )
    self.trainer._build_schedulers(resume_epoch=self.trainer._global_epoch)
```

**The reset creates brand-new optimizer objects via `optim.Adam(...)`
and `optim.SGD(...)`.** Fresh optimizers have empty `state` dicts; Adam's
`(m, v)` moments are zero; SGD's momentum buffers are zero. This is the
cleanest possible reset.

### Config verification

`configs/branchB1B2B3_mcts200.yaml`:
- `arena.revert_self_play_on_gate_failure: true` ✓ (revert enabled)
- `reset_optimizer_on_revert`: not set ⇒ uses default **True**
  (`supervisor.py:652-653`)

So the revert path AND the optimizer reset are both active. The reset
creates fresh optimizer instances. Optimizer state cannot survive revert.

### Conclusion: Mechanism 2 (carried optimizer state) — ❌ ELIMINATED

Falsified by code inspection. The revert path explicitly creates new
optimizer instances, which zeros Adam's `(m, v)` and SGD's momentum.
Iter 4 starting from iter_1's weights also starts from a freshly-zeroed
optimizer state — there is no carried direction from the rejected iter 3
training.

## 3. Updated drift mechanism priors

After both diagnostics:

| # | Mechanism | Pre-diag prior | Post-diag prior |
|---|---|---|---|
| 1 | Heuristic-weight annealing | ~30% | **~30-40%** |
| 2 | Optimizer state surviving revert | ~15% | **0%** (eliminated) |
| 3 | Buffer composition / mode collapse | ~5% | **0%** (eliminated) |
| 4 | Pure noise | ~50% | **~60-70%** |

**Bottom line:** the iter-3-4 drift is most likely just **noise** — three
data points each with SE ~2.5%, observing the sequence 52→50→45 in a
random walk around iter_1's true level is uncommon-but-not-rare. The
only remaining structural alternative is heuristic-weight annealing
(Mechanism 1), which requires a GPU re-run with
`heuristic_weight_end: 0.3` to test.

The "3 data points = noise" discipline rule from the investigation queue
is the right default. We should resist any future temptation to narrate
a 3-iter trajectory as a mechanism without first ruling out alternatives
the way we did here.

## 4. Implications for the next experiment (A4 + D1-partial)

**Buffer is safe to use for D1-partial pretraining:**
- Phase distribution is correct (76% MAIN_GAME — matches the underlying
  gameplay).
- No mode collapse signal that would invalidate the corpus.
- The 14.8% extreme-value samples might actually HELP the value head
  learn decisive outcomes faster — not a problem.
- After filtering dummy rows, effective corpus is ~38K samples (per
  buffer). Could combine both fixed-phase buffers (re-run #1 was killed
  mid-iter-3 so its buffer is ~30K-40K usable, re-run #2 is ~38K
  usable) for ~70-80K combined.

**No gate-fix needed before A4 + D1-partial:**
- A4 + D1-partial is supervised pretraining, no self-play loop, so no
  revert path is exercised.
- The optimizer-state and gate-override issues only affect runs that
  use the supervisor's self-play loop (B-family, D.1 v3 candidate).
- A4 + D1-partial can launch on the existing code without waiting for
  gate-tightening patches.

**Drift question is parked, not answered:**
- If A4 + D1-partial produces a meaningfully stronger model, we don't
  need to dig further into B1+B2+B3's iter-3-4 drift — the structural
  change (A4 / D1) is doing the work.
- If A4 + D1-partial also lands NOT_STRONGER, we may want to test
  Mechanism 1 (heuristic_weight_end=0.3) as a follow-up since it's the
  only remaining structural drift mechanism that hasn't been ruled out.
